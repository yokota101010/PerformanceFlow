# システム開発憲法 (System Development Constitution)

本憲法は、ドメインの本質を映し出すデータモデルの設計をすべての開発の出発点とし、堅牢で一貫性のあるシステムを構築するための共通規範である。

## 1. アーキテクチャ標準 (Architecture Standards)

### 1.1 論理定義 (普遍的アーキテクチャ原則)
- **アーキテクチャパターン**: ドメイン駆動設計（DDD）およびクリーンアーキテクチャの原則を採用する。
- **レイヤー構造と依存ルール**: アプリケーションは以下の3層で構成し、依存の方向は常に **「外側から内側（インフラストラクチャ → アプリケーション → ドメイン）」** の一方向とする。内側が外側の具体的な技術に依存してはならない。
  1. **ドメイン層 (Domain Layer)**: ビジネスルール、エンティティ、値オブジェクト、および **リポジトリのインターフェース（抽象的なデータアクセス契約）** を定義する。特定のフレームワークや外部ライブラリに一切依存しない、純粋なプログラミング言語のロジックのみで記述する。
  2. **アプリケーション層 (Application Layer)**: アプリケーションのユースケース（ビジネスプロセスの流れ）を実装する。UIが呼び出すための抽象インターフェース（UseCase/Query）と、その具象サービスをここに定義する。
  3. **インフラストラクチャ層 (Infrastructure Layer)**: ドメイン層のリポジトリインターフェースの具象実装（ブラウザのデータ永続化など）に加え、アプリケーション層 of ユースケースを呼び出す外部のインターフェース（Web UIなど）の実装を提供する。

### 1.2 物理実装 (本プロジェクトにおける具体的適用)
本プロジェクトにおいては、上記の論理構造を以下の技術スタックおよびリポジトリ構成にマッピングして実装する。

| 論理コンポーネント | 物理実装・使用技術 / 対象ファイル | 実行環境・配置位置 |
| :--- | :--- | :--- |
| **開発言語** | TypeScript | 共通 |
| **ドメイン層 (モデル・リポジトリ定義)** | 純粋なTypeScriptインターフェース・クラス・関数 | ブラウザプロセス内 |
| **アプリケーション層 (ユースケース・サービス)** | 純粋なTypeScriptクラス・関数・インターフェース | ブラウザプロセス内 |
| **インフラストラクチャ層 (UI・永続化実装)** | Reactコンポーネント、LocalStorageリポジトリ、インメモリリポジトリなど | ブラウザ（UI画面、ブラウザ内でのデータ保持・計算など） |
| **データモデルの正本 (SoT)** | `.specify/memory/domain-model.md` | リポジトリ内（ドキュメント正本） |
| **システム全体の鳥瞰図 (Overview)**| `.specify/memory/crud-matrix.md` | リポジトリ内（全体地図・CRUD図） |

## 2. データモデル・結果整合性の実現方針 (Data Model & Eventual Consistency)

### 2.1 論理定義 (普遍的整合性原則)
- **データモデル（エンティティ）駆動の原則**: 本システムは、ドメインエンティティ（データモデル）の設計をすべての開発の出発点とする。新しい機能の仕様策定および実装は、必ず対象となるデータの構造とビジネスルールの定義から開始しなければならない。画面（UI）の要件や特定のユースケースの都合によって、エンティティの整合性や本質的なデータ構造を歪めることを禁止する。
- **操作の一元管理**: ドメインオブジェクトの追加・変更・削除は、状態の不整合を防ぐため、必ず対応するドメイン層の「リポジトリインターフェース」を経由して実行しなければならない。
- **単一トランザクション・単一集約の原則**: 一度のユースケース実行（単一のビジネスユースケース）において、変更（書き込み）を許容する集約インスタンスは**必ず1つのみ**とする。複数の集約にまたがる同時変更は、境界の曖昧化やデッドロック等の原因となるため禁止する。
- **集約間の結果整合性の許容とプル型同期（オンデマンド）**: 異なる集約間、あるいは集約とそこから派生する集計・サマリデータ（導出データ）との間の整合性は、同一トランザクションでの即時同期を要求せず、結果整合性を許容する設計とする。
  - 具体的には、データ更新時は更新対象の単一集約のみを直接変更し、関連する集約やサマリデータは即時同期しない。代わりに、それらのデータを表示・利用するユースケース（画面やAPIクエリ）が起動されたタイミングで、真実の源泉（Source of Truth）となる一次データから最新状態をオンデマンドで算出し、必要に応じて読み取り専用の永続化モデルやキャッシュへ書き戻す「プル型（オンデマンド再計算・ライトバック）」のアプローチを採用する。
  - **将来的な結果整合性の拡張方針**: システムの規模拡大や非同期処理、他システム連携などの要件が発生し、プル型の再計算ではパフォーマンスや要件を満たせなくなった場合は、ドメインイベント（Domain Events）の発行とイベントディスパッチャーによるイベント駆動（プッシュ型同期）のアーキテクチャへと段階的に移行する。現時点での早期の導入は避ける。
- **エンティティの識別子（ID）採番原則**: エンティティ（特に集約ルート）の一意な識別子（ID）の生成および採番ロジックは、ドメイン層（UUIDなど）で決定するか、あるいはリポジトリインターフェースに `nextIdentity(): ID` などの契約を定義して解決すること。アプリケーションサービス層のインライン処理で、リポジトリ内の全データを走査して `maxNo + 1` のような手動計算・採番を行うことを禁止する（同時実行や整合性破壊を避けるため）。
- **値オブジェクトの完全置換原則**: 値オブジェクトは完全な不変（Immutable）とし、生成後の部分的なプロパティ変更を禁止する。値の一部を更新したい場合は、親エンティティにおいて新しい値オブジェクトのインスタンスを生成し、プロパティごと「丸ごと差し替える（完全置換）」こと。
- **ドメインロジックのドメイン層カプセル化の原則**: エンティティおよび値オブジェクトは、単なるデータの保持用コンテナ（データ構造体・インターフェース）であってはならない。`.specify/memory/domain-model.md` に定義されているビジネスルール、制約条件の検証（バリデーション）、および導出項目（計算項目）の算出ロジックは、すべてドメイン層のオブジェクト（クラスのコンストラクタやメソッド）の内部にカプセル化しなければならない。これらのロジックをアプリケーションサービス層やインフラストラクチャ層（リポジトリ）に分散・露出させることを禁止する。**また、複数集約の境界にまたがるビジネスルールや、外部の状態（他の集約など）に依存するビジネスルールについても、アプリケーション層で直接検証してはならない。検証に必要な外部データを引数として受け取るドメインエンティティのメソッド、または専用のドメインサービス（Domain Service）としてドメイン層にロジックをカプセル化し、アプリケーション層は必要なデータを取得してドメイン側に渡す「調整役」のみを担うこと。**
- **データモデル正本の絶対的真実性 (Single Source of Truth of Domain Schema)**: システムの整合性を司る「データモデル正本（ドメイン・スキーマ定義）」は、データ構造、整合性制約、ビジネスルール、および属性導出（計算）ロジックに関する唯一の絶対的な真実の源（Single Source of Truth）である。個別機能の仕様決定プロセス（Feature Specification）が、この正本に定められた数値の正負ルール（プラス・マイナスの制約）、許容範囲、またはライフサイクルルールと矛盾するローカル仕様を定義・上書きすることは一切許されない。
- **個別仕様におけるビジネスルールの「完全継承」 (No Information Loss in Feature Spec)**: 個別機能仕様書を作成・更新する際、該当機能が読み書き（C/R/U/D）するデータ項目に関連する、正本上のビジネスルール（更新タイミング、検証式、導出ロジック、正負の境界条件など）を省略・縮小することなく、完全に機能要件（FR）または受入基準（AC）へ継承（転記）しなければならない。「詳細はスキーマを参照」といった抽象的な省略記述は、実装時のビジネスロジック欠落（Information Loss）を招くため禁止する。

### 2.2 物理実装 (本プロジェクトにおける具体的適用)
本プロジェクトのブラウザ完結型環境において、上記の整合性を以下のように実現する。
- **データモデル正本および全体地図への準拠フロー**: 新機能の追加やリファクタリングの際、AIエージェントは必ず「1.2 物理実装」で指定された**データモデルの正本ファイル**および**システム全体の鳥瞰図ファイル**を最初に読み込み、システム全体の画面境界やスキーマ、型表現に厳格に従わなければならない。データモデルや全体設計の変更が必要な場合は、個別機能の変更に先立ち、まず当該ファイルの修正から開始する。なお、正本ファイル末尾に埋め込まれているデータモデルツール用のJSONメタデータ（`SCHEMA_DESIGNER_METADATA`）は開発上のロジックとは無関係であるため、解析時に無視して差し支えない。
- **画面分離とオンデマンド集計・同期による結果整合性の担保（物理的適用）**:
  - 論理的な「単一トランザクション・単一集約の原則」および「結果整合性」を物理的に満たすため、データ変更画面（入力・編集）とデータ集計表示画面（ダッシュボード・サマリ）を完全に分離する。
  - **画面起動時のプル同期（本プロジェクトにおける適用）**: 本プロジェクトのローカル永続化構成においては、サマリデータや月次残高の更新は、画面を開いたとき（または明示的にリロードが実行されたとき）のクエリ/ユースケース処理において、一次データを保持するリポジトリから最新データを取得して再計算し、結果整合性を担保する。
  - **データ表現**: サマリデータは状態（State）としてメモリ上に永続キャッシュせず、集計画面の起動時やユーザーによる明示的な更新トリガー（最新化ボタン等）の実行時に、リポジトリから取得した生のデータ配列に対して関数型操作（フィルタリング・畳み込み演算）を行うことで、その都度動的に算出（オンデマンド集計）する。
- **ブラウザローカル永続化のライフサイクル**:
  - アプリケーションが実行するデータの書き込み操作（Command）は、インメモリ上の状態更新と同時に、速やかに `LocalStorage` などのブラウザストレージへ同期（自動保存）されなければならない。
  - アプリケーション起動時には、`LocalStorage` に保存されたデータをリポジトリの初期状態として自動的にロードし、前回のデータ状態を復元すること。
- **データ型・構造マッピングとクラス化**:
  - 数値表現（`INT`, `DECIMAL` など） -> `number`
  - 文字列・日時表現（`VARCHAR`, `DATE`, `TIMESTAMP` など） -> `string`
  - 真偽値表現（`BOOLEAN`） -> `boolean`
  - **エンティティのクラス化と制約検証の強制**:
    - `.specify/memory/domain-model.md` の「3. テーブル定義」に定義されている各エンティティ（集約ルートおよび集約メンバー）は、TypeScriptの `class` として具現化して実装する。
    - テーブル定義のカラム型として直接使用されていない値オブジェクト定義（スキーマ内で独立定義されているが、テーブル定義のカラムの型に採用されていないもの）については、TypeScriptクラスとしての実装および使用を省略して差し支えない。
    - その場合、該当カラムのデータは上記のプリミティブ対応型（`number` や `string` 等）として扱い、付随する制約条件（「ビジネスルール・制約条件」や「カラム定義・ビジネスルール（制約）」等）は、オブジェクト生成時の不正な状態を防ぐため、該当エンティティクラスのコンストラクタ（または静的ファクトリメソッド）内での例外スロー（`throw new Error(...)`）によって強制検証すること。
    - 各テーブル定義の「導出項目」に定義されたロジック（例: 取引後残高、月末残高、月次集計データなど）は、エンティティクラスの読み取り専用ゲッター（`get` 属性）または計算メソッドとしてカプセル化し、外部サービス層でのインライン再計算や、単純な値の直接上書きを禁止する。
    - **複数集約にまたがるビジネスルールの適用例**:
      - 本プロジェクトにおける「月次確定済みの年月に対する取引操作（追加・修正・削除）の制限」は、取引明細集約が月次確定（サマリ）集約の状態に依存する、複数集約の境界をまたぐビジネスルールである。
      - この検証ロジックは、アプリケーションサービス（`TransactionInputService`）に直接実装するのではなく、`TransactionDetail` 等のエンティティクラスに外部データ（`MonthlyTransactionSummary` の一覧など）を引数として受け取る検証メソッド（例：`TransactionDetail.validateNotClosed(time, summaries)`）を定義してドメイン層にカプセル化する。または、専用のドメインサービス（`DomainService`）を定義して実装しなければならない。

## 3. コーディング＆運用標準 (Coding & Operational Standards)

### 3.1 論理定義 (普遍的規約原則)
- **命名による役割明示**: 識別子（変数、関数、クラス、インターフェースなど）は、アーキテクチャ上の役割や責務が名前から一目で判別できる規則を設けて命名すること。
- **不変性（Immutability）の担保**: 予期せぬ副次作用（サイドエフェクト）によるバグを抑止するため、オブジェクトや配列の状態を変更する際は、既存のインスタンスを直接書き換えず、新しいオブジェクト・配列を生成して割り当てるアプローチを徹底すること。

### 3.2 物理実装 (本プロジェクトにおける具体的適用)
TypeScript言語および本プロジェクトの運用において、上記の原則を以下のように具体化する。
- **命名ケース**: 変数名・関数名は `camelCase`、クラス名・インターフェース名は `PascalCase` を使用する。
- **ドメインオブジェクト（値オブジェクト・エンティティ）の配置と構築**:
  - **エンティティ (Entities / Projections)**:
    - クラス化されない読取専用エンティティやサマリデータ等のインターフェース定義（`MonthlyBalance` 等）は、`src/domain/models/types.ts` に配置すること。
    - 具象エンティティクラス（`Account`、`Category` 等）は、それぞれ個別のファイル（例: `src/domain/models/Account.ts`）に配置すること。
  - **値オブジェクト (Value Objects)**:
    - 値オブジェクトのインターフェース定義（`Money`、`Address` 等）は、`src/domain/value-objects/types.ts` に配置すること。
    - 具象値オブジェクトクラス（`Money`、`Address`）は、それぞれ個別のファイル（例: `src/domain/value-objects/Money.ts`）に配置すること。
  - 外部で定義された `interface` と整合性を保つため、ドメインクラスは `implements` を用いてこれらの型契約を満たすこと。
  - すべてのプロパティは基本的に不変とし、`readonly` 修飾子を付与すること。状態変化が発生する操作は、新しいインスタンスを複製して返すメソッド（イミュータブルパターン）として実装することを推奨する。
  - **インポートの効率化とモジュール境界の確立（バレルファイルの活用）**:
    - 各ドメインディレクトリ（`src/domain/models/`、`src/domain/value-objects/` 等）の `types.ts` は、純粋なデータ契約（`interface`）のみを定義し、具象クラスのインポート・再エクスポートを行ってはならない。
    - 各ディレクトリのエントリポイントとなるバレルファイル `index.ts` を配置し、公開するクラス（各個別ファイル）および型（`types.ts`）をすべてそこで集約して再エクスポートすること。
    - アプリケーション層やインフラ層などの外部クラスからこれらドメインオブジェクトをインポートする際は、個別のファイルパス（例: `../../domain/value-objects/Money.ts`）を直接指定せず、ディレクトリパス（例: `../../domain/value-objects`）を介してインポートすることを推奨する。これにより、将来的なファイル分割やフォルダ構成の変更に伴うインポート元の修正を不要にする（カプセル化）。
- **リポジトリインターフェースの命名と配置**:
  - ドメイン層で定義するリポジトリインターフェースの命名は、末尾に `Repository` を付与する（例: `TransactionRepository`）。
  - インターフェースファイルは `src/domain/repositories/` のようなドメイン層の配下に配置する。
  - *命名効果*: データ永続化の技術的な関心事（インメモリやDB等）をドメインから完全に隔離するとともに、データ保存/復元を担うモックや具象リポジトリを一目で識別可能にする。
- **ドメインサービスの命名と配置**:
  - 複数集約にまたがるビジネスロジックを記述するドメインサービスの命名は、末尾に `DomainService` を付与する（例: `AssetCalculationDomainService`）。ファイルは `src/domain/services/` 配下に配置する。
  - *命名効果*: エンティティや値オブジェクトの枠に収まらないドメインロジックを集約し、共通監査トレースや計算ロギングなどのアスペクト（AOP）適用を容易にする。
- **アプリケーション層（UseCase・Query・Service）の命名と配置**:
  - **ユースケース（書き込み系抽象インターフェース）**: 命名は末尾に `UseCase` を付与する（例: `TransactionInputUseCase`）。ファイルは `src/application/usecases/` 配下に配置する。
  - **クエリ（読み込み系抽象インターフェース）**: 命名は末尾に `Query` を付与する（例: `MonthlyDashboardQuery`）。ファイルは `src/application/usecases/` 配下に配置する.
  - **アプリケーションサービス（書き込み系具象実装）**: 命名は末尾に `Service` を付与する（例: `TransactionInputService`）。ファイルは `src/application/services/` 配下に配置する。
    - *命名効果*: トランザクション制御や共通例外ハンドリングなどのアスペクト（AOP）を、「Service」の実行前後へ一括適用しやすくする。
  - **クエリサービス（読み込み系具象実装）**: 命名は末尾に `QueryService` を付与する（例: `MonthlyDashboardQueryService`）。ファイルは `src/application/queries/` 配下に配置する。
    - *命名効果*: キャッシュ制御やパフォーマンス監査ログなどのアスペクト（AOP）を、「QueryService」の実行前後へ一括適用しやすくする。
  - **コマンド / DTO（ユースケース入力データ転送オブジェクト）**: 命名は末尾に `Command` を付与する（例: `RegisterIncomeExpenseCommand`）。
    - *命名効果*: コマンドの受信タイミングで、共通入力バリデーション（単項目検証）やデバッグ用の入力値ロギングを一元的・自動的に実行しやすくする。
- **インフラストラクチャ層の命名と配置**:
  - `infrastructure` 配下のフォルダは、技術的な関心事（ui, persistence）で分割したフラットなディレクトリ構成とする（`api` ディレクトリは作成しない）。
  - リポジトリ具象実装（Persistence）: `src/infrastructure/persistence/` 配下に配置し、具象クラス名は具体的なデータストア種別を冠して命名する（例: `InMemoryTransactionRepository` や `LocalStorageTransactionRepository`）。
  - **依存関係の解決とリポジトリ・レジストリ（重要）**:
    - テスト用（インメモリ）と本番用（LocalStorage）の実装を一元的に切り替えるため、`src/infrastructure/persistence/RepositoryRegistry.ts` を導入する。
    - UIコンポーネント、テストコード、他のリポジトリ具象実装など、いかなるコードにおいても、リポジトリの具象クラス（例: `new LocalStorageTransactionRepository()`）を直接インスタンス化することを禁止する（直接の `new` 禁止）。
    - 常に `RepositoryRegistry.getTransactionRepository()` のようにレジストリの静的メソッドまたはインスタンスを経由し、ドメイン層で定義されたリポジトリインターフェース型（例: `TransactionRepository`）としてインスタンスを取得しなければならない。
    - なお、インフラストラクチャ層ではないアプリケーションサービス（*Service、*QueryServiceなど）の具象クラスについては、UIコンポーネントやテストコード等において直接インスタンス化（`new`）することを許可する（その際、依存するリポジトリは `RepositoryRegistry` 経由で取得して注入する）。
    - テストや環境に応じたインスタンスの切り替えロジックは、この `RepositoryRegistry.ts` の内部（環境変数や静的初期化の制御など）にのみカプセル化する。
    - **将来的なインフラ抽象化の拡張性**: 将来的に外部API連携やシステムクロック等の抽象化・差し替えが必要になった場合は、このレジストリを `InfrastructureRegistry` 等に拡張して一元管理する。
  - **Web UI（UI画面・コンポーネント）**: `src/infrastructure/ui/` 配下に配置する。
    - 入力フォームや編集用のコンポーネントは末尾に `Form` を付与する（例: `TransactionForm.tsx`）。
    - 参照表示用の画面全体や一覧表示は末尾に `View` を付与する（例: `TransactionHistoryView.tsx`）。
    - *命名効果*: UIコンポーネントの役割（書き込み操作画面か、読み取り表示画面か）を明確にし、データ変更範囲（Command制約など）の自己検証を保ちやすくする。
- **出力言語の統一**: AIエージェントが生成するすべてのドキュメント（`spec.md`、`plan.md`、`tasks.md` など）、およびソースコード内のコメント・摘要は、原則として**日本語**で記述する。
- **不変性の強制**: エンティティや値オブジェクトの型定義において、すべてのプロパティに `readonly` 修飾子を付与すること。また、配列を扱う場合は `readonly T[]` または `ReadonlyArray<T>` を使用し、破壊的メソッド（`push`, `splice` 等）の呼び出しをコンパイルレベルで抑止すること。

## 5. コード品質標準 (Code Quality Standards)

### 5.1 プログラムコードの品質基準
- **静的型の厳格な適用**: 静的型付けが可能な言語環境において、コンパイラやリンターの型チェックを回避・無効化する記述を原則禁止とする（※TypeScriptにおける `any` の使用や `@ts-ignore` によるスキップの禁止）。型の特定が困難な場合は、適切な抽象化やドメイン固有の型定義を構築して解決すること。
- **例外およびエラーハンドリングの堅牢性**: 捕捉した例外やエラーを、単なるログ出力やコメントアウトのみで処理したと見なす（事実上の握りつぶし）実装を禁止する。エラー発生時は、呼び出し元への適切な伝播（再スロー）、またはコンテキストに応じた明示的なエラー状態（Result/Either型、または固有のエラーオブジェクト）の返却を行うこと。
- **マジックナンバーおよび生の文字列リテラルの排除**: ビジネスルールや制御ロジックに関わる数値・文字列を、コード内に生のままハードコードすることを禁止する。これらはすべて、意味のある命名を行った定数、列挙型（Enum）、または構成定義として抽出し、一元管理すること。

### 5.2 テストコードの品質基準
- **外部仕様と振る舞いの検証（リファクタリング耐性）**: テストは内部の具体的な実装詳細（非公開メソッドの呼び出し等）ではなく、公開されたインターフェース（外部仕様）に対する入力と出力の「振る舞い」を対象に記述すること。内部構造のリファクタリングによってテストコードが破壊されない設計を維持すること。
- **正本シードデータ（初期データ）の強制活用**:
  - *論理定義*: 単一集約の更新テストにおいて、更新対象外である参照先の集約（マスタや前提レコード）を個々のテストコードで捏造せず、正本で定義された共通サンプル（シードデータ）からロードして使用することで、テスト対象集約の検証に集中しデータ不整合を防ぐ。また、テスト品質（境界値や異常系の網羅）を担保するため、マスタの全区分（資産・負債、収入・支出）や主要な境界シナリオを表現する十分なサンプルインスタンスが正本に含まれていることを保証する。不足している場合は、実装・テストコードの作成に先立ち、まずデータモデル正本（spec.md）のシードデータ自体を拡張・整備すること。
  - *物理実装*: 自動テストの記述において初期状態をモックまたはインメモリ層にセットアップする際は、「1.2 物理実装」で指定されたデータモデル正本ファイル（`.specify/memory/domain-model.md`）の各テーブル定義に記載されている「AI検証用テストデータ (Seed Data)」のレコード群（特定の年月、マスタコード、残高、明細情報）を忠実に再現した共通フィクスチャを使用すること。
- **論理パスと境界値の網羅**: 外部仕様が定義するすべての条件分岐、例外経路（異常系）、および入力や状態の境界値（エッジケース）を検証するテストケースを必ず網羅すること。コードの有無によらず、仕様が求める全経路の検証を担保する。これらを達成するため、上記で定めた「正本シードデータ」が十分なバリエーションを持っている必要があり、テスト実装時の型安全チェック（静的解析によるundefined警告等）やテスト実行時の例外（マスタ不在エラー等）、およびカバレッジ測定結果における未踏パスの発見によってシードデータの不足を検知し、テストカバレッジ測定（網羅率の数値）を通してデータ不足度合いを定量的に評価した上で、速やかにデータモデル正本（spec.md）のシードデータを拡張しなければならない。
- **客観的な検証指標（カバレージ）の意識**: 外部仕様ベースのテストを通じて、結果として主要なビジネスロジックの実行経路（命令網羅・分岐網羅等）が十分に網羅（目安として90%以上）されている状態を目指すこと。テスト漏れの定量的な抑止指標として用いる。
- **テストコードの実装と継続性**: 実装されたテストケースは、プレースホルダーや省略記述（「後ほど実装」等のコメント）を用いず、すべて自動実行可能な具象コードとして記述すること。

## 8. ドキュメント設計・レビュー標準 (Document & Review Standards)

### 8.1 論理定義 (普遍的ドキュメント原則)
- **可読性・理解性の最大化**: AIエージェントが生成する仕様書（`spec.md`）や設計・計画書（`plan.md`）は、人間がレビュー・検証する際の知的負荷を最小化するよう設計されなければならない。情報が整理されず、巨大な文章の塊（Wall of Text）になっているドキュメントの生成を禁止する。
- **仕様の振る舞い（Behavior）駆動記述**: 受入基準（Acceptance Criteria）やビジネスルールは、曖昧な自然言語による手続き記述ではなく、テストケースへ1対1で対応するGiven-When-Then（BDD）形式に沿って構造化して記述すること。
- **視覚的理解（Visual First）**: 処理フロー、インターフェースの結合関係、およびドメイン状態の遷移は、テキスト説明に加えてMermaidダイアグラム等の図を強制的に挿入し、構造を一目で把握できるようにすること。
- **重要度情報の強弱制御**: レビュアーが見落ってはならない重要なビジネス上の制約、例外仕様、または実装のトレードオフは、一般的な文章と明確に区別し、視覚的なアラートとして強調表示すること。
- **ユビキタス言語のマッピング明文化**: ビジネス用語（日本語）とプログラム内部のコードシンボル（英語）の対応関係をドキュメントの冒頭に辞書として明示し、双方のレビューにおける用語の不一致を排除すること。
- **思考の連鎖（Chain of Thought: CoT）による事前自己検証の義務化**:
  - ドキュメント出力やファイル変更の実行前に、AIは自立した思考の連鎖を用いて「出力構造と前提ルールにエラーがないか」のチェックを内省的・段階的に実行しなければならない。
  - 特に、Markdownリンクのタグ閉じ・ネストの崩れ、Mermaidダイアグラム内の括弧（`()` や `[]`）の閉じ忘れや予約語衝突による構文エラー、および「データモデル正本（`spec.md`）の型・制約の意図せぬ省略や上書き」を事前に検知・修正する思考ステップを踏むこと。

### 8.2 物理実装 (本プロジェクトにおける具体的適用)
本プロジェクトの Spec Kit テンプレートおよび自動生成ワークフローにおいて、以下のルールを強制適用する。
- **Given-When-Then フォーマットの適用**:
  - `spec.md` 内の各機能要件および受入基準は、以下の書式で箇条書き記述する。
    - **Given**: [前提となるシステム状態・初期データ]
    - **When**: [ユーザーまたは外部システムからの操作・トリガー]
    - **Then**: [期待されるドメイン状態の変化・画面上の表示変化・出力値]
- **Mermaidによるシーケンス・状態図の挿入**:
  - `plan.md` のアーキテクチャ説明および `spec.md` のライフサイクルルールには、必ず `mermaid` 記法のコードブロックを用いて処理フロー（Sequence Diagram）や状態遷移（State Diagram）を可視化すること。
  - **Mermaidエラー防止の物理ガード**: Mermaid内のテキストラベル中に括弧 `()` や `[]` を含める場合は、構文破壊を防ぐために必ずダブルクォーテーションで囲むこと（例: `A["処理(引数)"]`）。
- **GitHub Alertsによる強調表現の制限と適用**:
  - 重要な注意事項やエラー制約は、GitHub Flavored Markdown の Alerts（`> [!IMPORTANT]`、`> [!WARNING]`、`> [!CAUTION]`）で記述すること。ただし、多用すると効果が薄れるため、1つの大項目につき最大2つまでに抑えること。
- **ユビキタス言語辞書の掲載**:
  - `spec.md` の「2. 仕様詳細」等のセクション冒表に、本機能で扱う業務概念の用語対訳表を markdown のテーブル形式で配置する（例: `| 業務上の用語 | コード内の英語表記 | 説明 |`）。
- **折りたたみタグ（`<details>`）の活用による縦長スクロールの抑制**:
  - 技術的な詳細（TypeScriptのインターフェースコード、Viteの設定、巨大なJSONペイロード等）は、以下のように `<details>` タグで囲んでデフォルトで非表示とすること。
    ```html
    <details>
    <summary>詳細なインターフェース・コード（クリックで開く）</summary>

    ```typescript
    // 詳細コード
    ```
    </details>
    ```
- **出力直前「自己検証（CoT Verification）チェックリスト」の実行（強制）**:
  - ドキュメントを新規作成・更新して書き込む際、AIエージェントは以下のチェックポイントについて思考空間（Chain of Thought）内で検証し、すべて適合していることを確認してからファイル書き込み（Tool Call）を行うこと。
    1. **Markdownリンク整合性**: `[テキスト](URL)` の閉じ括弧が抜けていないか、コードシンボルのリンクに余分なバックティックス（`[` や `]` 内への `` ` `` の混入）が含まれていないか。
    2. **Mermaid構文整合性**: 図のノード定義、接続方向（`-->`）に構文破綻がないか。
