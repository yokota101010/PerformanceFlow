## User Scenarios & Testing *(mandatory)*

### User Story 1 - 案件一覧の表示と初期データ投入 (Priority: P1) 🎯 MVP

登録されている案件（プロジェクトごとの案件情報）の一覧が画面上にテーブル形式で表示され、初回起動時には2件 of シードデータが自動的にロードされて表示される。

**Why this priority**: 案件情報の参照は、その後の追加・編集・削除、および案件作業明細（アサイン）などの他の後続処理すべての前提となるため。

**Independent Test**: LocalStorageやメモリ上のデータが空の状態でアプリを起動した際、初期シードデータとして 案件1（Ａソフト開発支援）および案件2（Ｂエンジニアリング開発支援）の2件がプロジェクト名および期間とともに自動的に一覧に表示されること。

**Acceptance Scenarios**:

1. **Given** アプリケーション初回起動時（LocalStorageに案件データが存在しない状態）、**When** 案件一覧画面を表示したとき, **Then** 以下のシードデータが自動投入され、案件IDの昇順（AJ001〜AJ002）でテーブルに表示されること。
   - プロジェクト: `PJ001`（Aシステム開発プロジェクト）, 案件ID: `AJ001`, 案件名: `案件1: Ａソフト開発支援`, 開始日: `2026-08-15`, 終了日: `2026-11-15`
   - プロジェクト: `PJ001`（Aシステム開発プロジェクト）, 案件ID: `AJ002`, 案件名: `案件2: Ｂエンジニアリング開発支援`, 開始日: `2026-09-13`, 終了日: `2026-10-31`
2. **Given** すでに案件が登録されている状態、**When** 案件一覧画面を表示したとき、**Then** 登録されているすべての案件が、紐づくプロジェクト名、期間とともにテーブル形式で表示されること。

### User Story 2 - 新しい案件の追加登録 (Priority: P1)

プロジェクトを選択し、案件名、開始日、終了日を入力して、自動採番された案件IDで新規登録できる。同一プロジェクト内での同名案件の登録は重複エラーとしてブロックされる。

**Why this priority**: 新たな案件受注に伴うアサインや発注契約を行うための事前マスタ登録として必要なため。

**Independent Test**: 登録フォームにてプロジェクトを選択し、案件名、期間を入力して登録した際、IDがプロジェクト単位で自動採番（プロジェクト内の既存IDの最大値+1）されて一覧に追加されること。

**Acceptance Scenarios**:

1. **Given** プロジェクト `PJ001`、案件名 `案件3: Ｃシステム開発支援`、開始日 `2026-10-01`、終了日 `2026-12-31` を入力し、**When** 登録を実行したとき、**Then** 案件ID `AJ003`（プロジェクトPJ001内での最大連番+1）で登録され、一覧に追加表示されること。
2. **Given** 異なるプロジェクト `PJ002` に対して初めて案件を追加する際、**When** 登録を実行したとき、**Then** 案件ID `AJ001`（プロジェクトPJ002内での初期値）で登録されること。
3. **Given** 入力値の前後にスペースが含まれる案件名を入力し、**When** 登録を実行したとき、**Then** 前後のスペースが自動的にトリミング（除去）されて登録されること。
4. **Given** 案件名が空欄、または開始日が終了日より未来の日付（例: 開始日 2026-11-01、終了日 2026-10-31）を入力し、**When** 登録を実行したとき、**Then** バリデーションエラーが表示され、登録が拒否されること。
5. **Given** 同一プロジェクト `PJ001` 内にすでに `案件1: Ａソフト開発支援` が登録されている状態、**When** 同一プロジェクトに同名の案件名を入力して登録を実行したとき、**Then** 「この案件名は同一プロジェクト内にすでに登録されています。」と警告エラーが表示され、登録が拒否されること（異なるプロジェクト間であれば同名案件の存在は許容する）。

### User Story 3 - 案件情報の編集・更新 (Priority: P2)

登録済みの案件の案件名、開始日、終了日を編集して更新できる。ただし、親となるプロジェクトIDの変更はできない。更新時も同一プロジェクト内での同名案件の重複は禁止される。

**Why this priority**: 案件の期間延伸や名称変更に追従できるようにするため。

**Independent Test**: 既存案件の編集画面を開き、情報を修正して保存した際、一覧画面に更新後の情報が即座に反映されること。

**Acceptance Scenarios**:

1. **Given** `AJ001` の編集画面を開き、終了日を `2026-12-31` に変更し、**When** 保存を実行したとき、**Then** 変更が反映され、一覧の `AJ001` の終了日が `2026-12-31` と表示されること。
2. **Given** 案件名を空欄にするか、開始日・終了日の順序が不正な日付を入力し、**When** 保存を実行したとき、**Then** バリデーションエラーが発生し、更新が拒否されること。
3. **Given** プロジェクト `PJ001` 内に `AJ001`（案件1）と `AJ002`（案件2）が存在する状態、**When** `AJ002` の編集で案件名を `案件1: Ａソフト開発支援`（AJ001と同名）に変更して保存を実行したとき、**Then** 重複エラーメッセージが表示され、更新が拒否されること。

---

### User Story 4 - 案件の物理削除 (Priority: P2)

不要になった案件データをマスタから完全に抹消（物理削除）する。ただし、案件作業明細（アサイン契約実績データ）など、他集約から参照されている案件の削除は整合性の観点からブロックする。

**Why this priority**: 誤登録された案件データの削除を可能にしつつ、関連契約実績データの整合性を維持するため。

**Independent Test**: 他集約から参照されていない案件が正常に削除できること、および参照されている案件の削除が警告メッセージとともに拒否されること。

**Acceptance Scenarios**:

1. **Given** 関連する実績データ（`案件作業明細.案件ID`）が存在しない案件に対して、**When** 削除を実行したとき、**Then** 確認ダイアログの承認後、マスタからデータが物理削除（完全に消滅）されること。
2. **Given** 関連する実績データ（`案件作業明細.案件ID`）レコードが少なくとも1件存在する案件に対して、**When** 削除を実行したとき、**Then** 警告メッセージ「この案件はアサイン実績（案件作業明細）から参照されているため削除できません。」が表示され、削除がブロックされること。

---

### Edge Cases

- **連番IDの上限値**: プロジェクト内の案件発行が `AJ999` に達した状態でさらに登録を試みた場合、システムは「案件IDの発行上限に達しました。」というエラーを出して登録を制限しなければならない。
- **入力トリミングによる空欄**: 案件名にスペース（全角・半角）のみを入力して登録しようとした場合、トリミングによって空文字と判定され、「案件名は必須です。」とエラーになること。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは、親となるプロジェクトIDと案件IDの組み合わせ（複合キー）によって案件を一意に識別しなければならない。
- **FR-002**: システムは、案件IDを `AJnnn`（AJ固定、3桁連番）の形式で自動採番しなければならない。
- **FR-003**: 採番範囲について、案件IDはプロジェクトIDごとに連番をクリア（リセット）し、同一プロジェクトIDの内部で `AJ001` から順番に採番しなければならない。別のプロジェクトIDとの間で案件ID（例: AJ001）が重複することは許容される。
- **FR-004**: システムは、案件名の前後のスペース（全角・半角）を保存およびバリデーションの直前に自動的にトリミングしなければならない。
- **FR-005**: 案件名は必須入力とし、最大長は255文字としなければならない。
- **FR-006**: システムは、開始日および終了日を必須入力とし、`YYYY-MM-DD` 形式でなければならない。
- **FR-007**: システムは、「開始日 <= 終了日」でなければならないという整合性を検証し、開始日が終了日より未来である場合は登録・更新を拒否しなければならない。
- **FR-008**: システムは、同一プロジェクト内での案件名の重複について、重複登録および重複名への更新を禁止しなければならない。同一プロジェクト内にすでに同じ案件名が存在する場合、重複エラーメッセージ「この案件名は同一プロジェクト内にすでに登録されています。」を表示し処理を拒否しなければならない。異なるプロジェクト間での同名案件の存在は許容する。
- **FR-009**: システムは、案件の削除処理を行う際、対象の（プロジェクトID、案件ID）のペアが `案件作業明細` にて参照されているかをチェックし、関連レコードが少なくとも1件存在する限り、一律で削除をブロックし警告メッセージ「この案件はアサイン実績（案件作業明細）から参照されているため削除できません。」を表示しなければならない。
- **FR-010**: 削除の制限ルールに抵触しない案件データは、LocalStorageおよびメモリ内ストアから完全に消滅する「物理削除」を行わなければならない。

### Key Entities *(include if feature involves data)*

- **案件 (Case / ProjectAssignment)**:
  - 案件を表す集約ルート。
  - 属性:
    - `プロジェクトID` (複合主キー, 外部キー: `プロジェクト.プロジェクトID`)
    - `案件ID` (複合主キー, `AJnnn` 形式)
    - `案件名` (文字列, 255文字以下)
    - `開始日` (日付)
    - `終了日` (日付)

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザーは案件の追加・編集・削除の主要アクションを、それぞれの操作画面で迷うことなく3ステップ以内で実行できる。
- **SC-002**: 登録を実行した際、新しい案件IDが重複や欠番再利用なく、1秒以内に自動採番されて一覧に即座に反映される。
- **SC-003**: 削除制限対象の案件を削除しようとした場合、1秒以内に画面に適切な日本語の警告メッセージが表示され、データが保護されること。
- **SC-004**: ビルドされた成果物が、完全にスタンドアローン（`file://` スキーム）でブラウザ上で手動実行および検証できること。

## Assumptions

- 案件IDの連番は最大999までサポートする（AJ999）。これを超える登録は当面のスコープ外とする。
- データの初期ロードは、アプリケーションの起動時（またはリロード時）にLocalStorageから読み込むこととする。
- 親プロジェクトが削除されるのを防ぐため、すでにプロジェクトマスタ側で「案件が紐づいているプロジェクトの削除はブロックする」仕様が適用されている前提とする。
