# Quickstart Validation Guide: F05 案件管理

本ドキュメントは、「F05 案件管理」機能の実装後、開発者や検証担当者が機能要件（CRUD、プロジェクト単位採番、同一プロジェクト内重複禁止、日付順序制限、削除制限など）をローカル環境で検証するための手順および検証シナリオを定義する。

---

## 1. 準備と実行手順 (Setup & Execution)

### 前提条件 (Prerequisites)
* Node.js v18 以上
* npm パッケージマネージャ

### アプリケーションの起動と検証方法
以下のいずれかの方法でアプリケーションを起動し、検証を行います。

#### 方法A: スタンドアローン形式でのビルド・起動 (推奨)
1. 以下のコマンドでアセットをビルドします：
   ```bash
   npm run build
   ```
2. ビルド完了後、生成された `dist/index.html` 単一ファイルをブラウザで直接ダブルクリックして開きます。

#### 方法B: 開発用ローカルサーバーでの起動
1. 開発サーバーを起動します：
   ```bash
   npm run dev
   ```
2. 起動後、ブラウザで [http://localhost:5173](http://localhost:5173) を開きます。

#### 自動テストの実行
```bash
npm run test
```
Vitest による自動検証テストを実行し、すべてのテストがパスすることを確認します。

## 2. 受入検証シナリオ (Validation Scenarios)

### シナリオ1: アプリ初回起動時の初期データ自動投入検証
* **検証内容**: 永続化ストレージ（LocalStorage）が空の状態でアプリを初めて開いた際、シードデータ（2件）が自動投入され、プロジェクト名、期間が表示されること。
* **手順**:
  1. ブラウザのデベロッパーツールを開き、「アプリケーション（Application）」タブから `LocalStorage` のデータを全消去（クリア）する。
  2. 画面をリロードする。
  3. ヘッダーの「案件管理」タブをクリックする。
  4. 案件一覧画面に、以下の2件の案件情報が正しく表示されていることを確認する。
     - **プロジェクト**: `Aシステム開発プロジェクト` (PJ001)
     - **案件ID**: `AJ001`
     - **案件名**: `案件1: Ａソフト開発支援`
     - **期間**: `2026-08-15` 〜 `2026-11-15`
     <br />
     - **プロジェクト**: `Aシステム開発プロジェクト` (PJ001)
     - **案件ID**: `AJ002`
     - **案件名**: `案件2: Ｂエンジニアリング開発支援`
     - **期間**: `2026-09-13` 〜 `2026-10-31`

### シナリオ2: 新規登録とプロジェクト単位採番の検証
* **検証内容**: 案件の追加時に、プロジェクト単位で個別に案件IDが自動採番されること。
* **手順**:
  1. 登録フォームにて、プロジェクトドロップダウンから `Aシステム開発プロジェクト`（PJ001）を選択する。
  2. 案件名に `案件3: Ｃシステム開発支援` と入力し、開始日に `2026-10-01`、終了日に `2026-12-31` を入力して「登録」をクリックする。
  3. 一覧に **案件ID**: `AJ003`（PJ001内での最大値+1）で登録されていることを確認する。
  4. 次に、登録フォームにて、別のプロジェクト（例: `基幹システム刷新プロジェクト` (PJ002)）を選択する。
  5. 案件名に `プロジェクト初期設計支援` と入力し、開始日に `2026-08-01`、終了日に `2026-09-30` を入力して「登録」をクリックする。
  6. プロジェクト PJ002 に対して最初の案件が、**案件ID**: `AJ001`（PJ002内のクリア・初期値）として新しく採番され登録されていることを確認する。

### シナリオ3: 同一プロジェクト内での同名案件重複ブロック検証
* **検証内容**: 同一プロジェクト内に全く同じ名称の案件を重複登録しようとした場合に、警告されて拒否されること。
* **手順**:
  1. 登録フォームにて、プロジェクト `Aシステム開発プロジェクト`（PJ001）を選択する。
  2. すでに登録されている `案件1: Ａソフト開発支援` と全く同じ案件名を入力し、期間を入力して「登録」をクリックする。
  3. 「この案件名は同一プロジェクト内にすでに登録されています。」と警告エラーが表示され、登録が拒否されることを確認する。
  4. プロジェクトを `基幹システム刷新プロジェクト`（PJ002）に変更し、同じ案件名 `案件1: Ａソフト開発支援` を入力して「登録」をクリックする。
  5. プロジェクトが異なるため、正常に登録され、一覧に表示されることを確認する。

### シナリオ4: 日付整合性バリデーションの検証
* **検証内容**: 開始日が終了日より未来である場合にバリデーションエラーとなること。
* **手順**:
  1. 登録フォームにて、開始日に `2026-11-01`、終了日に `2026-10-31`（開始日より過去）を入力し、「登録」ボタンをクリックする。
  2. 「開始日は終了日以前の日付で入力してください。」とバリデーション警告が表示され、登録が拒否されることを確認する。

### シナリオ5: 案件情報の編集・更新の検証（親プロジェクト不変・重複禁止）
* **検証内容**: 既存の案件情報を変更できること。また、編集フォームにおいて親プロジェクトIDは変更できないように保護されていること。更新時も同一プロジェクト内の名称重複チェックが動作すること。
* **手順**:
  1. 一覧から `PJ001` - `AJ002` の「編集」ボタンをクリックする。
  2. フォームにデータがロードされ、プロジェクト選択フィールドが「無効化（disabled）」されていることを確認する。
  3. 案件名を `案件2: Ｂエンジニアリング開発支援 (延伸)` に変更し、終了日を `2026-12-15` に変更して「保存」ボタンをクリックする。
  4. 正常に保存され、一覧画面の `AJ002` の案件名および終了日が即座に更新されることを確認する。
  5. 再度 `AJ002` の編集を開き、案件名を `案件1: Ａソフト開発支援`（AJ001と同名）に変更して「保存」をクリックする。
  6. 「この案件名は同一プロジェクト内にすでに登録されています。」と重複警告が表示され、更新が拒否されることを確認する。

### シナリオ6: 案件の物理削除とアサイン紐づき削除ブロックの検証
* **検証内容**: アサイン実績（案件作業明細）が存在する案件は削除がブロックされ、参照のない案件は正常に物理削除されること。
* **手順**:
  1. 一覧画面で `AJ001` または `AJ002`（初期シードデータで、アサイン実績がある案件）の「削除」ボタンをクリックする。
  2. 「この案件はアサイン実績（案件作業明細）から参照されているため削除できません。」というエラー警告が表示され、削除がブロックされることを確認する。
  3. 先ほど登録した `AJ003`（実績の紐づいていない案件）の「削除」ボタンをクリックし、確認ダイアログでOKを押す。
  4. 一覧から `AJ003` が正常に消滅することを確認する。
  5. LocalStorage内のレコードから `AJ003` のデータが物理的に抹消されていることを確認する。
