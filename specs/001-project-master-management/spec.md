# Feature Specification: F01 プロジェクトマスタ管理

**Status**: Draft

## Clarifications

### Session 2026-07-18

- Q: プロジェクト名の重複登録を許可するか？ → A: 重複登録を禁止する（新規登録・更新時に同名のプロジェクトが既に存在すればエラーとする）。
- Q: プロジェクト削除時のデータ処理方針はどうするか？ → A: 物理削除（LocalStorage/メモリのデータストアから完全にレコードを削除する）。
- Q: 初期データ（Seed Data）の自動投入ポリシーはどうするか？ → A: アプリ初回起動時に、シードデータ（PJ001: 「次世代基幹システム開発プロジェクト」）を自動的に投入し、登録済みの状態でスタートする。
- Q: プロジェクトIDの自動採番における欠番の取り扱いはどうするか？ → A: 欠番は再利用せず、常に「現在登録されている最大の数値 + 1」で採番する。
- Q: プロジェクト名入力時における前後の空白（スペース）の取り扱いはどうするか？ → A: 自動でトリミングを行う（入力値の前後の空白を除去し、スペースのみの入力は「空欄（必須バリデーションエラー）」として処理する）。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - プロジェクト一覧の表示 (Priority: P1)

ユーザーは、システムに登録されているプロジェクトの一覧をテーブル形式で確認できる。

**Why this priority**: システムの基本情報であるプロジェクトを参照するために最優先で必要となる機能。

**Independent Test**: プロジェクト一覧画面を開き、登録されているプロジェクトがリスト形式で正しく表示されることを検証できる。

**Acceptance Scenarios**:

1. **Given** 登録済みのプロジェクト（PJ001: 「次世代基幹システム開発プロジェクト」など）が存在する状態において、
   **When** プロジェクト一覧画面を表示したとき、
   **Then** プロジェクトID「PJ001」、プロジェクト名「次世代基幹システム開発プロジェクト」がテーブルに表示されること。
2. **Given** プロジェクトが1件も登録されていない状態において、
   **When** プロジェクト一覧画面を表示したとき、
   **Then** 「プロジェクトが登録されていません」などの旨を示すメッセージが表示されること。

---

### User Story 2 - プロジェクトの新規登録 (Priority: P1)

ユーザーは、プロジェクト名を入力して新しいプロジェクトをシステムに登録できる。

**Why this priority**: プロジェクトを追加するための基本機能。

**Independent Test**: 新規登録フォームに値を入力して保存し、一覧画面に自動採番されたIDと共に新しいプロジェクトが表示されることで検証できる。

**Acceptance Scenarios**:

1. **Given** プロジェクト一覧画面または登録用フォームにおいて、
   **When** プロジェクト名（例: 「新規製品開発プロジェクト」）を入力し、登録を実行したとき、
   **Then** 新しいプロジェクトが登録され、IDが自動採番（例: `PJ002`）されて一覧に追加されること。
2. **Given** プロジェクト登録画面において、
   **When** プロジェクト名を空欄のまま登録を実行したとき、
   **Then** エラーメッセージ（「プロジェクト名は必須です」など）が表示され、登録が拒否されること。
3. **Given** すでに「次世代基幹システム開発プロジェクト」という名称のプロジェクトが登録されている状態において、
   **When** 新たに「次世代基幹システム開発プロジェクト」というプロジェクト名で登録を実行したとき、
   **Then** 重複エラーメッセージが表示され、登録が拒否されること。
4. **Given** プロジェクト登録画面において、
   **When** プロジェクト名に「  （半角スペースのみ）」と入力して登録を実行したとき、
   **Then** 前後のスペースがトリミングされ、必須チェックにより「プロジェクト名は必須です」というエラーメッセージが表示され、登録が拒否されること。

### User Story 3 - プロジェクト名の変更・更新 (Priority: P1)

ユーザーは、登録済みのプロジェクトの名称を編集・変更できる。

**Why this priority**: 誤入力の修正や名称変更に対応するための基本機能。

**Independent Test**: 特定のプロジェクトの編集画面を開き、名称を変更して保存した後、一覧や詳細画面で変更後の名称が反映されていることで検証できる。

**Acceptance Scenarios**:

1. **Given** プロジェクト「PJ001」（名称: 「次世代基幹システム開発プロジェクト」）が存在する状態において、
   **When** プロジェクト名を「次世代基幹システム開発プロジェクト改」に変更して保存したとき、
   **Then** 変更が適用され、一覧画面で「PJ001」の名称が「次世代基幹システム開発プロジェクト改」に更新されていること。
2. **Given** プロジェクト編集画面において、
   **When** プロジェクト名を空欄にして保存を実行したとき、
   **Then** エラーメッセージが表示され、更新が拒否されること。
3. **Given** すでに「PJ001: 次世代基幹システム開発プロジェクト」と「PJ002: 新規製品開発プロジェクト」が登録されている状態において、
   **When** 「PJ002」の名称を「次世代基幹システム開発プロジェクト」に変更して保存を試みたとき、
   **Then** 重複エラーメッセージが表示され、更新が拒否されること。
4. **Given** プロジェクト「PJ001」の編集画面において、
   **When** プロジェクト名に「  次世代基幹システム開発プロジェクト改  」と前後にスペースを入れて保存を実行したとき、
   **Then** 前後のスペースがトリミングされ、「PJ001」の名称が「次世代基幹システム開発プロジェクト改」とスペースなしで更新されること。

### User Story 4 - プロジェクトの削除 (Priority: P1)

ユーザーは、不要になったプロジェクトをシステムから削除できる。ただし、該当プロジェクトに紐づく「案件」が存在する場合は削除が制限される。

**Why this priority**: 不要データのクリーンアップに必要だが、データの整合性を守るための制約を伴う。

**Independent Test**: 案件の紐づいていないプロジェクトが正常に削除できること、および案件が紐づいているプロジェクトの削除が適切にブロックされることで検証できる。

**Acceptance Scenarios**:

1. **Given** 案件が1件も紐づいていないプロジェクト（例: `PJ002`）が存在する状態において、
   **When** 削除を実行したとき、
   **Then** プロジェクトが一覧から正常に削除されること。
2. **Given** 既に `案件` に紐づいているプロジェクト（例: `PJ001` が案件 `AJ001` から参照されている状態）において、
   **When** 削除を実行したとき、
   **Then** 削除がブロックされ、「このプロジェクトには案件が登録されているため削除できません」というエラーメッセージが表示されること。

---

### Edge Cases

- **プロジェクトIDの自動採番上限**: 
  自動採番の形式は `PJnnn`（`nnn` は001からの連番）である。連番が999に達した場合の新規登録時、システムは適切なエラーメッセージ（「プロジェクトIDの発行上限に達しました」など）を表示し、新規登録を安全に拒否する。
- **不正な文字長入力**:
  プロジェクト名に255文字を超える極端に長い文字列を入力して登録・更新を試みた場合、バリデーションエラーを発生させ、入力を拒否する。
- **同名プロジェクトの登録・更新試行**:
  すでに存在するプロジェクト名と同じ名称を設定しようとした場合、一意性バリデーションエラーを発生させ、処理を拒否する。
- **スペースのみの入力**:
  全角および半角スペースのみでプロジェクト名を登録しようとした場合、バリデーション前にトリミングされ、最終的に空文字判定となるため登録を拒否する。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは登録されているプロジェクトの一覧を表示する画面を提供しなければならない（表示項目：プロジェクトID、プロジェクト名）。
- **FR-002**: プロジェクトの新規作成時および名称編集時、プロジェクト名は必須入力（空文字不可）とし、最大255文字までの制限を設けなければならない。この際、システムは値の前後の空白（全角・半角スペース）を自動的にトリミングし、トリミング後の値に対して空欄チェックおよび重複チェックを実行しなければならない。
- **FR-003**: 新規作成時のプロジェクトIDは、データストア内の既存プロジェクトの最大連番（ID数値部分）に1を加えた値を `PJnnn`（3桁ゼロ埋め）形式でシステムが自動採番しなければならない。この際、削除処理等によって生じた途中の欠番は再利用しない。また、ユーザーによる手動入力・編集は不可としなければならない。
- **FR-004**: プロジェクト名の編集時、プロジェクトIDは不変（編集不可）でなければならない。
- **FR-005**: 削除対象のプロジェクトIDが `案件` テーブルの `プロジェクトID` カラムで参照されている場合、システムは削除処理をブロックし、ユーザーに対して参照が存在する旨のエラーを表示しなければならない。
- **FR-006**: 削除対象のプロジェクトIDが他のリソースから参照されていない場合、システムはLocalStorageやメモリ上のデータストアから該当レコードを完全に消去（物理削除）しなければならない。
- **FR-007**: プロジェクトの登録・編集・削除処理（Command系インターフェース）は、プロジェクト集約（`プロジェクト` テーブルのみ）の操作に閉じなければならない（憲法4.1条のCommand制約）。同一画面から同時に案件や他の集約を変更・登録・削除する機能を混在させてはならない。
- **FR-008**: システムは、新規登録時および名称編集時に同一プロジェクト名が既に登録されていないか重複チェックを行い、重複している場合は保存処理を拒否してエラーを表示しなければならない。
- **FR-009**: システムは、LocalStorage等の永続化データが存在しない初回起動時に、シードデータ（プロジェクトID: PJ001, プロジェクト名: 次世代基幹システム開発プロジェクト）をデータストアに自動投入しなければならない。

### Key Entities *(include if feature involves data)*

- **プロジェクト (Project)**:
  - `プロジェクトID` (`VARCHAR(255)`、PK): 自動採番される一意の識別子。形式は `PJnnn`（例: PJ001）。
  - `プロジェクト名` (`VARCHAR(255)`、必須、一意): プロジェクトの識別名称。同一名称の重複登録は不可。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザーはプロジェクトの登録・一覧参照・編集・削除の基本操作（CRUD）をブラウザUI上で完結できること。
- **SC-002**: 案件が紐づいているプロジェクトを削除しようとした際、100%確実に削除がブロックされ、適切な警告メッセージが表示されること。
- **SC-003**: Command系処理（登録・更新・削除）実行時に、憲法の変更制約に従い、プロジェクト集約以外のドメインオブジェクト（`案件` など）への意図しない変更処理がAPIやインフラ層で一切呼び出されないこと。
- **SC-004**: 同名のプロジェクトが複数システムに登録されるケースが検証時に発生しないこと（一意性の保証）。
- **SC-005**: 削除されたプロジェクトデータが、ブラウザの永続化ストレージ（LocalStorage等）から完全に抹消されていること。
- **SC-006**: アプリケーションを初回起動した直後に、プロジェクト一覧画面に `PJ001` が自動的に表示されていること。
- **SC-007**: 前後に意図しないスペースが含まれた状態で登録・更新を行っても、データベースおよび一覧画面にはスペースが取り除かれた名称で登録されていること。

## Assumptions

- **永続化と実行環境**: 本プロジェクトはSPAで動作し、バックエンドサーバーは存在しないため、プロジェクトデータはブラウザの `LocalStorage` またはインメモリに永続化される。
- **依存データの検証**: プロジェクトの削除可能性を判定する際、ローカル環境の `案件リポジトリ` 等から当該 `プロジェクトID` を持つレコードを検索して判定を行う。
- **多言語対応**: UI表示およびエラーメッセージはすべて日本語で提供される（憲法3.2条のドキュメント標準に準拠）。
- **削除処理方式**: データの削除は物理削除で行い、論理削除用のフラグ管理は導入しない。
- **初回初期データ投入**: データ永続化先にデータが存在しない状態での起動時に限り、システム規定 of 初期マスタデータとしてPJ001が自動生成される。
- **ID自動採番方針**: プロジェクト削除などにより欠番が生じても、採番時には常に「過去最大の連番 + 1」を使用し、欠番の再利用は行わない。
- **入力値のトリミング**: プロジェクト名に対するバリデーションおよび保存処理の直前に、前後のスペースはシステム側で自動的に除去（トリミング）される。
