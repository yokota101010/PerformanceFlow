# Quickstart Validation Guide: F06 案件作業契約（明細）管理

本ドキュメントは、「F06 案件作業契約（明細）管理」機能の実装後、開発者や検証担当者が機能要件（自動計算、プロジェクト単位採番、期間重複・隙間チェック、自動終了日計算、削除制限など）をローカル環境で検証するための手順および検証シナリオを定義する。

---

## 1. 準備と実行手順 (Setup & Execution)

### アプリケーションの起動
1. 以下のコマンドでアセットをビルドします：
   ```bash
   npm run build
   ```
2. ビルド完了後、生成された `dist/index.html` 単一ファイルをブラウザで直接開くか、`npm run dev` でローカルサーバーを起動し [http://localhost:5173](http://localhost:5173) を開きます。

#### 自動テストの実行
```bash
npm run test
```
Vitest による自動検証テストを実行し、すべてのテストがパスすることを確認します。

---

## 2. 受入検証シナリオ (Validation Scenarios)

### シナリオ1: アプリ初回起動時の初期データ自動投入と売上・粗利率の自動計算検証
* **検証内容**: LocalStorageが空の状態でアプリを初めて開いた際、シードデータ4件（WK001〜WK004）が自動ロードされ、売上、粗利、粗利率（四捨五入して小数点2桁表示）が自動計算されていること。
* **手順**:
  1. ブラウザのデベロッパーツールを開き、`LocalStorage` のデータをクリアする。
  2. 画面をリロードし、ヘッダーの「案件作業明細（アサイン契約）」タブをクリックする。
  3. 一覧に以下の4件のシードデータが、売上や粗利率等の計算値とともに正しく表示されていることを確認する。
     * **WK001**: 売上 `8,000,000` 円、製造原価 `5,242,000` 円、粗利 `2,758,000` 円、粗利率 `0.34`
     * **WK002**: 売上 `8,000,000` 円、製造原価 `5,215,000` 円、粗利 `2,785,000` 円、粗利率 `0.35`
     * **WK003**: 売上 `1,400,000` 円、製造原価 `2,490,000` 円、粗利 `-1,090,000` 円、粗利率 `-0.78`
     * **WK004**: 売上 `1,400,000` 円、製造原価 `2,490,000` 円、粗利 `-1,090,000` 円、粗利率 `-0.78`

### シナリオ2: 終了日の自動計算および期間隙間チェックの検証
* **検証内容**: 新規登録時に開始日を入力した際、期間の順序関係から終了日が自動算出されること、および案件全体の期間との間に「隙間」が発生する場合はバリデーションエラーとなること。
* **手順**:
  1. 登録フォームにて、プロジェクト `PJ001`（Aシステム開発プロジェクト）、案件 `AJ001`（案件1、期間: 2026-08-15 〜 2026-11-15）を選択する。
  2. 登録されている明細は現在以下の状態：
     - WK001: 2026-08-15 〜 2026-09-30
     - WK002: 2026-10-01 〜 2026-11-15
  3. 新しい明細を追加するために、開始日に `2026-10-15` を入力し、工数・単価を入れて「登録」をクリックする。
  4. 登録を実行すると、WK002の期間と重複し、かつ結合した期間の隙間が矛盾するため、期間整合性エラーが発生し登録が拒否されることを確認する。
  5. 次に、既存のWK002を一旦削除（※アサイン実績がないテストデータを用意する等）するか、または期間を完全に分割カバーするように明細を再登録する。
     * 例：明細を3つに分割する場合：
       - 明細A: 2026-08-15開始 (終了日は自動で09-14)
       - 明細B: 2026-09-15開始 (終了日は自動で09-30)
       - 明細C: 2026-10-01開始 (終了日は自動で11-15 - 案件終了日)
     * これらが重複なく、かつ開始日から終了日まで隙間なく繋がっている場合のみ、正常に登録・更新が成功することを確認する。

### シナリオ3: プロジェクト単位の採番検証
* **検証内容**: アサイン明細のIDがプロジェクト単位で個別に自動採番（最大値+1）されること。
* **手順**:
  1. 登録フォームにて、別のプロジェクト（例: `基幹システム刷新プロジェクト` (PJ002)）の案件を選択する。
  2. 開始日、工数、単価を入力し登録を実行する。
  3. 登録された明細のIDが **WK001**（プロジェクトPJ002内のクリア初期値）として新しく採番され登録されていることを確認する。

### シナリオ4: 削除制限における参照制約ブロックの検証
* **検証内容**: 後続の実績データ（発注や社員工数等）から参照されている明細は削除がブロックされ、参照のない明細は物理削除できること。
* **手順**:
  1. 一覧画面で `WK001` または `WK002`（初期シードデータで、発注実績ORD001等が紐づいている明細）の「削除」ボタンをクリックする。
  2. 「この作業契約は発注実績、工数実績、またはその他経費実績から参照されているため削除できません。」というエラー警告が表示され、削除がブロックされることを確認する。
  3. 新しく追加した、実績データの紐づいていない明細の「削除」をクリックし、確認ダイアログでOKを押す。
  4. 一覧から正常に消滅することを確認する。
