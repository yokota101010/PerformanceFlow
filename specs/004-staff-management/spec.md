# Feature Specification: F04 要員マスタ管理

**Feature Branch**: `004-staff-management`

**Created**: 2026-07-18

**Status**: Draft

**Input**: User description: "/speckit-specify F04の開発を行う"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 要員一覧の表示と初期データ投入 (Priority: P1) 🎯 MVP

登録されている要員（パートナー企業の要員）の情報一覧が画面上にテーブル形式で表示され、初回起動時には4名のシードデータが自動的にロードされて表示される。

**Why this priority**: 要員情報が存在し一覧で確認できることは、登録・編集・削除などすべての後続マスタ管理操作の前提となる最重要機能であるため。

**Independent Test**: LocalStorageやメモリ上のデータが空の状態でアプリを起動した際、初期シードデータとして 坂本龍馬、高杉晋作、西郷隆盛、勝海舟の4名が所属会社名および単価とともに自動的に一覧に表示されること。

**Acceptance Scenarios**:

1. **Given** アプリケーション初回起動時（LocalStorageに要員データが存在しない状態）、**When** 要員一覧画面を表示したとき、**Then** 以下のシードデータが自動投入され、要員IDの昇順（MEM001〜MEM004）でテーブルに表示されること。
   - `MEM001`: 所属 `BP001`（Ａソフトウェア）, 氏名 `坂本龍馬`, 単価 `1,000,000` 円
   - `MEM002`: 所属 `BP001`（Ａソフトウェア）, 氏名 `高杉晋作`, 単価 `700,000` 円
   - `MEM003`: 所属 `BP002`（Ｂエンジニアリング）, 氏名 `西郷隆盛`, 単価 `850,000` 円
   - `MEM004`: 所属 `BP002`（Ｂエンジニアリング）, 氏名 `勝海舟`, 単価 `600,000` 円
2. **Given** すでに要員が登録されている状態、**When** 要員一覧画面を表示したとき、**Then** 登録されているすべての要員が、所属会社名および単価とともにテーブル形式で表示されること。

---

### User Story 2 - 新しい要員の追加登録 (Priority: P1)

要員の「氏名」、「所属会社（発注先）」、「単価（月額）」を入力・選択し、新規IDを自動採番して要員マスタに追加登録できる。

**Why this priority**: 新たに外部要員をプロジェクトにアサインする際、マスタへ要員を登録する必要があるため。

**Independent Test**: 登録フォームに氏名、所属会社、単価を入力して登録した際、IDが自動採番（既存IDの最大値+1）されて一覧に追加されること。

**Acceptance Scenarios**:

1. **Given** 氏名 `岡田以蔵`、所属会社 `BP001`、単価 `500,000` を入力し、**When** 登録を実行したとき、**Then** 要員ID `MEM005`（最大連番+1）で登録され、一覧に追加表示されること。
2. **Given** 入力値の前後にスペースが含まれる氏名を入力し、**When** 登録を実行したとき、**Then** 前後のスペースが自動的にトリミング（除去）されて登録されること。
3. **Given** 氏名が空欄、または単価に負数を入力し、**When** 登録を実行したとき、**Then** エラーメッセージが表示され、登録が拒否されること。
4. **Given** すでに登録されている要員名と同名（例: `坂本龍馬`）の氏名を入力し、**When** 登録を実行したとき、**Then** 新規要員ID `MEM005` で正常に重複登録され、同姓同名の2行が一覧に表示されること。

---

### User Story 3 - 要員情報の編集・更新 (Priority: P2)

登録済みの要員の「氏名」、「所属会社（発注先）」、「単価」を編集して更新できる。

**Why this priority**: 要員の単価改定や所属会社変更、または氏名誤入力の修正に追従できるようにするため。

**Independent Test**: 既存要員の編集画面を開き、情報を修正して保存した際、一覧画面に更新後の情報が即座に反映されること。

**Acceptance Scenarios**:

1. **Given** `MEM001` の編集画面を開き、単価を `1,200,000` に変更し、**When** 保存を実行したとき、**Then** 変更が反映され、一覧の `MEM001` の単価が `1,200,000` と表示されること。
2. **Given** 氏名を空欄にするか、単価に不正な値（負の数など）を入力し、**When** 保存を実行したとき、**Then** バリデーションエラーが発生し、更新が拒否されること。

---

### User Story 4 - 要員の物理削除 (Priority: P2)

不要になった要員のデータをマスタから完全に抹消（物理削除）する。ただし、発注（注文明細実績データ）など、他集約から参照されている要員の削除は整合性の観点からブロックする。

**Why this priority**: 誤登録された要員データの削除を可能にしつつ、契約実績データの整合性を維持するため。

**Independent Test**: 他集約から参照されていない要員が正常に削除できること、および参照されている要員の削除が警告メッセージとともに拒否されること。

**Acceptance Scenarios**:

1. **Given** 関連する実績データ（`注文明細.要員ID` または `月別要員工数サマリ.要員ID`）が存在しない要員に対して、**When** 削除を実行したとき、**Then** 確認ダイアログの承認後、マスタからデータが物理削除（完全に消滅）されること。
2. **Given** 関連する注文実績（`注文明細.要員ID`）または `月別要員工数サマリ` のレコードが少なくとも1件存在する要員に対して、**When** 削除を実行したとき、**Then** 警告メッセージ「この要員は実績データから参照されているため削除できません。」が表示され、削除がブロックされること。

---

### Edge Cases

- **連番IDの上限値**: IDの発行が `MEM999` に達した状態でさらに登録を試みた場合、システムは「要員IDの発行上限に達しました。」というエラーを出して登録を制限しなければならない。
- **入力トリミングによる空欄**: 氏名にスペース（全角・半角）のみを入力して登録しようとした場合、トリミングによって空文字と判定され、「氏名は必須です。」とエラーになること。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは、要員IDを `MEMnnn`（MEM固定、3桁連番）の形式で自動採番しなければならない。
- **FR-002**: 採番にあたっては、途中の欠番は再利用せず、常に「過去最大の連番 + 1」で採番しなければならない。
- **FR-003**: システムは、氏名の前後のスペース（全角・半角）を保存およびバリデーションの直前に自動的にトリミングしなければならない。
- **FR-004**: 氏名は必須入力とし、最大長は255文字としなければならない。
- **FR-005**: 単価は必須入力とし、`0` 以上の整数（月単価）でなければならない。
- **FR-006**: システムは、要員登録・更新時における氏名の重複について、同姓同名の要員の重複登録を許可しなければならない。氏名が同一であっても、自動採番される「要員ID」によって一意性を識別しなければならない。
- **FR-007**: システムは、要員の所属会社について、所属会社IDを必須入力とし、すでに「発注先マスタ」に登録されている発注先IDのみを選択・指定可能としなければならない。「所属なし（空欄）」での登録は拒否しなければならない。
- **FR-008**: システムは、要員の削除処理を行う際、対象の要員IDが `注文明細.要員ID`（注文明細実績）および `月別要員工数サマリ.要員ID` のいずれかから参照されているかをチェックし、関連レコードが存在する限り、一律で削除をブロックし警告メッセージを表示しなければならない。
- **FR-009**: 削除の制限ルールに抵触しない要員データは、LocalStorageおよびメモリ内ストアから完全に消滅する「物理削除」を行わなければならない。

### Key Entities *(include if feature involves data)*

- **要員 (Staff / Resource)**:
  - パートナー要員を表す集約ルート。
  - 属性:
    - `要員ID` (主キー, `MEMnnn` 形式)
    - `所属会社ID` (外部キー: `発注先.発注先ID`)
    - `氏名` (文字列, 255文字以下)
    - `単価` (数値, 0以上の整数)

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザーは要員の追加・編集・削除の主要アクションを、それぞれの操作画面で迷うことなく3ステップ以内で実行できる。
- **SC-002**: 登録を実行した際、新しい要員IDが重複や欠番再利用なく、1秒以内に自動採番されて一覧に即座に反映される。
- **SC-003**: 削除制限対象の要員を削除しようとした場合、1秒以内に画面に適切な日本語の警告メッセージが表示され、データが保護されること。
- **SC-004**: ビルドされた成果物が、完全にスタンドアローン（`file://` スキーム）でブラウザ上で手動実行および検証できること。

## Assumptions

- 要員IDの連番は最大999までサポートする（MEM999）。これを超える登録は当面のスコープ外とする。
- データの初期ロードは、アプリケーションの起動時（またはリロード時）にLocalStorageから読み込むこととする。
- 所属会社ID（FK）に紐づく発注先情報が削除されるのを防ぐため、すでに発注先マスタ側で「要員が所属している場合は発注先の削除をブロックする」仕様が適用されている前提とする。
