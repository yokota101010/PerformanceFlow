# Quickstart Validation Guide: F04 要員マスタ管理

本ドキュメントは、「F04 要員マスタ管理」機能の実装後、開発者や検証担当者が機能要件（CRUD、同姓同名、所属会社必須、削除制限など）をローカル環境で検証するための手順および検証シナリオを定義する。

---

## 1. 準備と実行手順 (Setup & Execution)

### 前提条件 (Prerequisites)
* Node.js v18 以上
* npm パッケージマネージャ

### アプリケーションの起動と検証方法
以下のいずれかの方法でアプリケーションを起動し、検証を行います。

#### 方法A: スタンドアローン形式でのビルド・起動 (推奨)
1. 以下のコマンドでアセットをビルドします：
   ```bash
   npm run build
   ```
2. ビルド完了後、生成された `dist/index.html` 単一ファイルをブラウザで直接ダブルクリックして開きます。

#### 方法B: 開発用ローカルサーバーでの起動
1. 開発サーバーを起動します：
   ```bash
   npm run dev
   ```
2. 起動後、ブラウザで [http://localhost:5173](http://localhost:5173) を開きます。

#### 自動テストの実行
```bash
npm run test
```
Vitest による自動検証テストを実行し、すべてのテストがパスすることを確認します。

## 2. 受入検証シナリオ (Validation Scenarios)

### シナリオ1: アプリ初回起動時の初期データ自動投入検証
* **検証内容**: 永続化ストレージ（LocalStorage）が空の状態でアプリを初めて開いた際、シードデータ（4名）が自動投入され、所属会社名、単価が表示されること。
* **手順**:
  1. ブラウザのデベロッパーツールを開き、「アプリケーション（Application）」タブから `LocalStorage` のデータを全消去（クリア）する。
  2. 画面をリロードする。
  3. ヘッダーの「要員マスタ」タブをクリックする。
  4. 要員一覧画面に、以下の4名の要員情報が正しく表示されていることを確認する。
     - `MEM001`: 所属 `Ａソフトウェア`, 氏名 `坂本龍馬`, 単価 `1,000,000` 円
     - `MEM002`: 所属 `Ａソフトウェア`, 氏名 `高杉晋作`, 単価 `700,000` 円
     - `MEM003`: 所属 `Ｂエンジニアリング`, 氏名 `西郷隆盛`, 単価 `850,000` 円
     - `MEM004`: 所属 `Ｂエンジニアリング`, 氏名 `勝海舟`, 単価 `600,000` 円

### シナリオ2: 新規登録とID自動採番の検証（同姓同名の許可を含む）
* **検証内容**: 新しい要員が自動採番されたIDで追加されること。所属会社は発注先マスタのリストから選択できること。また、同姓同名の重複登録が正常に許可されること。
* **手順**:
  1. 登録フォームにて、氏名に `岡田以蔵` と入力し、所属会社ドロップダウンから `Ａソフトウェア`（BP001）を選択、単価に `500,000` と入力し、「登録」ボタンをクリックする。
  2. 一覧画面に以下の通り追加されていることを確認する。
     - **要員ID**: `MEM005`（最大値 + 1）
     - **所属**: `Ａソフトウェア`
     - **氏名**: `岡田以蔵`
     - **単価**: `500,000` 円
  3. 再度登録フォームにて、全く同姓同名である `岡田以蔵`、所属 `Ｂエンジニアリング`（BP002）、単価 `550,000` と入力し、「登録」ボタンをクリックする。
  4. 重複エラーにならず正常に登録され、一覧に `MEM006` として `岡田以蔵` が追加されることを確認する（同姓同名の別人が識別可能）。

### シナリオ3: バリデーションと入力トリミングの検証
* **検証内容**: 氏名の前後の空白がトリミングされること。また、氏名空欄や単価の負数がエラーで弾かれること。
* **手順**:
  1. 登録フォームで、氏名に `  中岡慎太郎  `（前後にスペース）と入力し、所属 `Ａソフトウェア`、単価 `650,000` で登録を実行する。
  2. 登録完了後、一覧画面に表示された氏名が `中岡慎太郎`（スペースなし）になっていることを確認する。
  3. 登録フォームで、氏名にスペースのみを入力、または単価に `-100`（負数）を入力して「登録」をクリックする。
  4. 「氏名は必須です。」または「単価は0以上の整数で入力してください。」とエラーが表示され、登録が拒否されることを確認する。

### シナリオ4: 要員情報の編集・更新の検証
* **検証内容**: 既存の要員情報（氏名、所属、単価）を変更して保存できること。
* **手順**:
  1. 一覧から `MEM001` の「編集」ボタンをクリックする。
  2. フォームに `MEM001`（坂本龍馬）のデータがロードされることを確認する。
  3. 単価を `1,200,000`、所属を `Ｂエンジニアリング` に変更し、「保存」ボタンをクリックする。
  4. 正常に保存され、一覧上の `MEM001` の所属が `Ｂエンジニアリング`、単価が `1,200,000` 円に即座に更新されることを確認する。

### シナリオ5: 要員の物理削除と制限の検証
* **検証内容**: 契約実績（注文明細や工数サマリ）が紐づいている要員は削除が拒否され、紐づいていないものは物理削除されること。
* **手順**:
  1. 一覧画面で `MEM001`〜`MEM004`（初期シードデータで、注文実績等がある要員）の「削除」ボタンをクリックする。
  2. 「この要員は実績データから参照されているため削除できません。」というエラー警告が表示され、削除がブロックされることを確認する。
  3. 先ほど登録した `MEM005`（実績の紐づいていない要員）の「削除」ボタンをクリックし、確認ダイアログでOKを押す。
  4. 一覧から `MEM005` が正常に消滅することを確認する。
  5. LocalStorage内のレコードから `MEM005` のデータが物理的に抹消されていることを確認する。
