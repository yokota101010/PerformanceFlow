# Feature Specification: 社員工数実績入力 (employee-worktime-management)

**Feature Branch**: `008-employee-worktime-management`

**Created**: 2026-07-19

**Status**: Draft

**Input**: User description: "F08を開発する"

---

## 1. 概要 (Overview)
本機能は、自社社員がどの案件作業（アサイン）に対して、対象の年月に、どれだけの稼働時間（工数）を実績として費やしたかを入力・管理する機能である。
登録された工数実績は、ドメインスキーマに定義された計算式に基づいて自動的に「加工費」として算出され、案件アサイン側の「製造原価」計算へ動的にプル同期（結果整合）される。

### 1.1 ユビキタス言語対訳表
| 業務上の用語 | コード内の英語表記 | 説明 |
| :--- | :--- | :--- |
| 月別案件社員工数 | EmployeeWorkTime | 社員の案件・年月ごとの稼働時間実績を表すドメイン集約ルート。 |
| 月別社員工数サマリ | EmployeeWorkTimeSummary | 社員ごとの年月別の合計稼働時間を集計したドメイン集約。 |
| 作業契約ID | caseAssignmentId | 稼働対象となるアサイン契約の識別子。 |
| 社員ID | staffId | 稼働した自社社員の識別子。 |
| 年月 | targetMonth | 稼働が発生した対象年月 (YYYY-MM-01形式)。 |
| 作業時間 | workHours | 稼働した実績時間（時間単位、0〜200時間）。 |
| 加工費 | laborCost | 社員の時間単価と作業時間から算出される製造原価（社員労務費）。 |

## 2. ユーザーシナリオとテスト (User Scenarios & Testing)

### User Story 1 - 社員工数実績の一覧表示と合計サマリ (Priority: P1) 🎯 MVP
自社社員の稼働実績データの一覧を表示し、初期状態ではシードデータ（6件）がロードされる。また、画面下部に月別の合計作業時間（サマリ）が自動集計されて表示される。

* **Why this priority**: システム全体の収支・原価計算に直結する一次データであり、画面上での一覧性と整合性計算を確認できる最小の実用インクリメント（MVP）となるため。
* **Independent Test**: LocalStorageやメモリ上のデータが空の状態でアプリを起動し、「社員工数実績入力」画面を開いた際、初期シード明細 6件（WK001〜WK004に関するEMP001/EMP002のデータ）が正しい加工費、合計時間とともに一覧上に自動でロードされて表示されること。
* **Acceptance Scenarios**:
  1. **Given** データベースが初期状態（シード状態）のとき、**When** 実績入力ビューを開くと、**Then** 6件の実績レコードが表示され、加工費（トム・デマルコ: 9,000円/h × 160h = 1,440,000円 等）が正しく計算されて表示されていること。
  2. **Given** 実績一覧が表示されているとき、**When** 特定の年月を選択してフィルタリングを行うと、**Then** 該当の年月のレコードのみが絞り込まれ、サマリ合計時間がその月の稼働時間合計に連動して再計算されること。

---

### User Story 2 - 工数実績の新規登録 (Priority: P1)
対象の作業契約、社員、年月、および作業時間を指定して、新しい工数実績データを登録できる。

* **Why this priority**: 過去実績の登録や新しい要員稼働実績の入力を行うための必須CRUD機能であるため。
* **Independent Test**: 新規登録フォームに正常値を入力して登録した際、一意性制約エラーや期間外エラー、最大作業時間超過警告が働き、正常値のみがリポジトリへ登録され、一覧に即座に反映されること。
* **Acceptance Scenarios**:
  1. **Given** 新規登録ダイアログを開き、作業契約 `WK001`、社員 `EMP001`、年月 `2026-08`、作業時間 `150` を入力して登録すると、**Then** 重複がなく契約期間内（WK001は2026-08-15〜2026-09-30）であるため登録が成功し、一覧の合計値が加算されること。
  2. **Given** 登録ダイアログで、アサイン期間外の年月（例: `WK001` に対して `2026-07` などの期間外）を入力したとき、**When** 登録ボタンを押すと, **Then** 「選択された年月は作業契約の対象期間外です。」のエラーが表示され、保存がブロックされること。
  3. **Given** 同一の「作業契約 ＋ 社員 ＋ 年月」の組み合わせが既に登録されているとき、**When** 重複する組み合わせで登録しようとすると、**Then** 「同一の年月、作業契約、社員に対する工数実績が既に登録されています。」のエラーが発生して保存がブロックされること。

---

### User Story 3 - 工数実績の編集・更新 (Priority: P2)
一覧に表示されている登録済みの工数実績データを選択し、作業時間を編集・変更できる。

* **Why this priority**: 入力ミスや稼働計画変更に伴う時間実績の修正を可能にするため。
* **Independent Test**: 特定の行の「編集」ボタンを押し、作業時間を変更して保存した際、一覧画面上の「加工費」および合計時間が即座に再計算されて反映されること。
* **Acceptance Scenarios**:
  1. **Given** 登録済みの実績（トム・デマルコ, 160時間）の編集フォームを開き、作業時間を `120` に変更して保存すると、**Then** 更新が成功し、加工費が `1,080,000` 円（9,000円 × 120h）に再計算されて一覧に反映されること。
  2. **Given** 編集画面において、作業時間に `250`（200時間超）や `-10`（マイナス値）を入力したとき、**When** 保存を押すと、**Then** バリデーションエラーが作動し、保存がブロックされること。

---

### User Story 4 - 工数実績の物理削除 (Priority: P2)
不要になった工数実績データを物理削除できる。

* **Why this priority**: 不要データや誤入力データを完全にシステムから取り除くため。
* **Independent Test**: 削除を実行した際、一覧から即座に消去され、リポジトリ上からも物理的に消滅すること。
* **Acceptance Scenarios**:
  1. **Given** 実績一覧の特定の行の「削除」ボタンを押し、確認ダイアログで承認すると、**Then** リポジトリおよび画面一覧からレコードが完全に削除され、合計稼働時間からそのレコードの時間が減算されること。

---

### エッジケース (Edge Cases)
- **月次稼働上限サマリ警告**:
  - 同一社員の特定年月における合計稼働時間（`月別社員工数サマリ`）が 200時間 を超過するような登録・更新が行われた場合、システムは保存をブロックせず許可するが、画面上で警告（アラート）メッセージ「月間合計稼働時間が200時間を超過しています」を表示してユーザーに警告する。
- **削除制限制約**:
  - 本集約（工数実績）は他から物理参照されるFKの制限はありませんが、案件作業明細（アサイン契約 F06）側でアサインが削除される際の「工数実績から参照されているため削除できない」制約が正しく機能する（カスケード削除ではなく、削除ブロックの参照元データとなる）。

## 3. 要件 (Requirements)

### 機能要件 (Functional Requirements)
- **FR-001**: システムは、工数実績データ（`月別案件社員工数`）をブラウザの LocalStorage に自動永続化し、起動時に自動的にロードして復元しなければならない。
- **FR-002**: システムは、初回起動時に以下の 6件のシードデータを自動投入しなければならない。
  - `WK001`, `EMP001`, `2026-08-01`, `160` (加工費 1,440,000円)
  - `WK001`, `EMP001`, `2026-09-01`, `160` (加工費 1,440,000円)
  - `WK002`, `EMP001`, `2026-10-01`, `160` (加工費 1,440,000円)
  - `WK002`, `EMP001`, `2026-11-01`, `160` (加工費 1,440,000円)
  - `WK003`, `EMP002`, `2026-09-01`, `160` (加工費 1,280,000円)
  - `WK004`, `EMP002`, `2026-10-01`, `160` (加工費 1,280,000円)
- **FR-003**: 登録・更新時の作業時間は `0` 以上 `200` 以下の整数値でなければならない。
- **FR-004**: 対象の `年月` (月初日形式 `YYYY-MM-01`) は、アサインされた `案件作業明細` の期間（開始日・終了日）と部分的に重なる（月内にアサイン期間が1日でも含まれている）必要がある。
- **FR-005**: 工数実績登録時、社員はアサイン制限なく、システムに存在するすべての作業契約（作業契約ID）に対して自由に稼働実績時間を入力できる。
- **FR-006**: 同一社員の同一年月における複数アサインに跨る合計作業時間（月別社員工数サマリ）が 200時間 を超過した場合、保存自体は許可するが、UI上で警告（アラート）メッセージ「月間合計稼働時間が200時間を超過しています」を表示してユーザーに通知しなければならない。
- **FR-007**: 加工費（`laborCost`）は `単価 (社員単価)` × `作業時間` として動的に算出されなければならない。

## 4. 主要なエンティティ (Key Entities)

### `EmployeeWorkTime` (月別案件社員工数)
社員の案件（アサイン）ごとの月次稼働時間を表す集約ルート。
- `caseAssignmentId`: 案件作業明細ID (FK)
- `staffId`: 社員ID (FK)
- `targetMonth`: 対象年月 (YYYY-MM-01形式)
- `workHours`: 作業時間 (0〜200)
- `laborCost` (ゲッター): 加工費 (作業時間 × 社員単価)

### `EmployeeWorkTimeSummary` (月別社員工数サマリ)
社員ごとの月次合計作業時間を表す読取専用サマリ（結果整合）。
- `staffId`: 社員ID (FK)
- `targetMonth`: 対象年月 (YYYY-MM-01形式)
- `totalWorkHours` (ゲッター): 該当社員・年月におけるすべての `EmployeeWorkTime.workHours` の合計値。

## 5. 評価基準 (Success Criteria)

### 測定可能な成果 (Measurable Outcomes)
- **SC-001**: ユーザーは、工数実績の追加・更新・削除を3ステップ以内で画面から実行できること。
- **SC-002**: 登録された工数実績が保存された際、案件作業明細 (F06) ビューを開いた時に、対応する「製造原価」および「粗利」「粗利率」に金額がオンデマンドで正しく自動反映されること。
- **SC-003**: 200時間を超える無効な作業時間の入力や、アサイン対象外の年月の入力が、クライアント側で確実にブロックされ、詳細なエラーメッセージが表示されること。

---

## 6. 前提条件と前提 (Assumptions)
- 社員マスタ（F02）および案件作業アサインマスタ（F06）には、シードデータに対応する `EMP001`, `EMP002`, `WK001`〜`WK004` が既に存在していることを前提とする。
- 社員単価は `社員` テーブルの `単価` (時間単価) を使用する。
- 1人月の上限は200時間とし、これを一律の上限値とする。
