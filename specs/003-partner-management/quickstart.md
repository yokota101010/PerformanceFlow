# Quickstart Validation Guide: F03 発注先マスタ管理

本ドキュメントは、「F03 発注先マスタ管理」機能の実装後、開発者や検証担当者が機能要件（CRUD、重複エラー、削除制限など）をローカル環境で検証するための手順および検証シナリオを定義する。

---

## 1. 準備と実行手順 (Setup & Execution)

### 前提条件 (Prerequisites)
* Node.js v18 以上
* npm パッケージマネージャ

### アプリケーションの起動と検証方法
以下のいずれかの方法でアプリケーションを起動し、検証を行います。

#### 方法A: スタンドアローン形式でのビルド・起動 (推奨)
1. 以下のコマンドでアセットをビルドします：
   ```bash
   npm run build
   ```
2. ビルド完了後、生成された `dist/index.html` 単一ファイルをブラウザで直接ダブルクリック（またはドラッグ＆ドロップ）して開きます。ローカルサーバーの起動は不要です。

#### 方法B: 開発用ローカルサーバーでの起動
1. 開発サーバーを起動します：
   ```bash
   npm run dev
   ```
2. 起動後、ブラウザで [http://localhost:5173](http://localhost:5173) を開きます。

#### 自動テストの実行
```bash
npm run test
```
Vitest による自動検証テストを実行し、すべてのテストがパスすることを確認します。

## 2. 受入検証シナリオ (Validation Scenarios)

### シナリオ1: アプリ初回起動時の初期データ自動投入検証
* **検証内容**: 永続化ストレージ（LocalStorage）が空の状態でアプリを初めて開いた際、シードデータ（2件）が正しく自動投入され、発注先IDの昇順で表示されること。
* **手順**:
  1. ブラウザのデベロッパーツールを開き、「アプリケーション（Application）」タブから `LocalStorage` のデータを全消去（クリア）する。
  2. 画面をリロードする。
  3. ヘッダーの「発注先マスタ」タブをクリックする。
  4. 発注先一覧画面に、以下の発注先情報が正しく表示されていることを確認する。
     - `BP001`: Ａソフトウェア
     - `BP002`: Ｂエンジニアリング

### シナリオ2: 新規登録とID自動採番の検証
* **検証内容**: 新しい発注先が自動採番（最大値 + 1）されたIDで追加されること。また、重複した発注先名の登録がエラーでブロックされること。
* **手順**:
  1. 登録フォームにて、発注先名に `Ｃシステムズ` と入力し、「登録」ボタンをクリックする。
  2. 一覧画面に以下の通り追加されていることを確認する。
     - **発注先ID**: `BP003`（`BP002` の最大値 `2` に `+1` されたもの）
     - **発注先名**: `Ｃシステムズ`
  3. 再度登録フォームにて、すでに存在する発注先名である `Ａソフトウェア` と入力し、「登録」ボタンをクリックする。
  4. 「この発注先名はすでに登録されています。」とエラーメッセージが表示され、登録が拒否されることを確認する。

### シナリオ3: 入力値トリミングの検証
* **検証内容**: 入力前後のスペースが自動的に削除されること。
* **手順**:
  1. 登録フォームで、発注先名に `  Ｄネットワークス  `（前後にスペース）と入力して登録を実行する。
  2. 登録完了後、一覧画面に表示された名称が `Ｄネットワークス`（スペースなし）になっていることを確認する。
  3. 登録フォームで、発注先名にスペースのみ（例: `  `）を入力して登録を実行する。
  4. 「発注先名は必須です。」とエラーが表示され、登録できないことを確認する。

### シナリオ4: 発注先情報の編集・更新の検証
* **検証内容**: 既存発注先の名前の編集ができること。また、編集時にも重複名が制限されること。
* **手順**:
  1. 一覧から `BP001` の「編集」ボタンをクリックする。
  2. フォームに `BP001` のデータがロードされることを確認する。
  3. 名前を `Ｂエンジニアリング` （すでに存在する別発注先の名前）に書き換え、「保存」ボタンをクリックする。
  4. 「この発注先名はすでに登録されています。」とエラー警告が表示され、保存が拒否されることを確認する。
  5. 名前を `Ａソフトウェア株式会社` に書き換え、「保存」をクリックする。
  6. 正常に保存され、一覧上の `BP001` の名前が即座に `Ａソフトウェア株式会社` に更新されることを確認する。

### シナリオ5: 発注先の物理削除と制限の検証
* **検証内容**: 所属要員や発注実績が紐づいている発注先は削除が拒否され、紐づいていないものは物理削除されること。
* **手順**:
  1. 一覧画面で `BP001` または `BP002`（初期シードデータで、所属要員や発注実績がある取引先）の「削除」ボタンをクリックする。
  2. 「この発注先は他テーブルから参照されているため削除できません。」というエラー警告が表示され、削除がブロックされることを確認する。
  3. 先ほど登録した `BP003`（実績や要員所属の紐づいていない取引先）の「削除」ボタンをクリックし、確認ダイアログでOKを押す。
  4. 一覧から `BP003` が正常に消滅することを確認する。
  5. LocalStorage内のレコードから `BP003` のデータが物理的に抹消されていることを確認する。
