# Feature Specification: 発注（注文）管理 (Partner Order Management)

**Feature Branch**: `007-partner-order-management`

**Created**: 2026-07-18

**Status**: Draft

**Input**: User description: "F07の開発を行う"

本機能は、パートナー（発注先）に対する発注データおよび注文明細データの管理（CRUD）を可能にし、各アサイン（作業契約明細）に紐づく製造原価の動的な算出に寄与する。

## Clarifications

### Session 2026-07-18
- Q: 新規発注における「年月と発注先」の重複登録制限 → A: 同一の `作業契約ID` + `年月` + `発注先ID` の組み合わせでの重複登録を禁止し、バリデーションエラーとする。
- Q: 注文明細における要員の「所属会社（発注先）」一致バリデーション → A: 所属会社IDと発注先IDの完全一致をバリデーションで強制し、不一致の場合はエラーで保存を拒否する。

## 1. ユビキタス言語辞書

| 業務上の用語 | コード内の英語表記 | 説明 |
| :--- | :--- | :--- |
| 注文ID | `id` / `orderId` | 発注データの一意な識別子（形式：ORDnnn） |
| 発注 / 注文 | `PartnerOrder` | パートナー（発注先）に対する月次の発注契約データ |
| 注文明細 | `OrderDetail` | 発注データに紐づく、要員ごとの発注工数・単価データ |
| 作業契約ID | `caseAssignmentId` | 紐づく案件作業明細（アサイン契約）のID（形式：WKnnn） |
| 発注先ID | `partnerId` | 紐づくパートナー（発注先）のID（形式：BPnnn） |

| 合計発注額 | `totalAmount` | 注文明細における `発注額` の合計値（円） |

## 2. User Scenarios & Testing *(mandatory)*

### User Story 1 - 発注および注文明細の一覧表示と自動計算 (Priority: P1) 🎯 MVP

登録されている発注情報の一覧を画面表示し、発注を選択した際に紐づく注文明細（要員、工数、単価、発注額）を一覧テーブル形式で表示する。また、明細データから全体の合計工数・合計発注額が自動計算されて表示される。

**Why this priority**: パートナー発注データを可視化し、製造原価の構成要素（発注額）を正しく計算するためのMVPとして最も重要。

**Independent Test**: 
初期シードデータ 6件（ORD001〜ORD006）およびそれらに紐づく注文明細が、金額・工数自動計算とともに一覧表示されること。

**Acceptance Scenarios**:

1. **Given**: 初期シードデータ 6件が登録された状態。
   **When**: 発注管理画面を開いたとき。
   **Then**: 注文ID `ORD001`〜`ORD006` の一覧テーブルが表示され、対応する作業契約ID、発注先名、年月、合計工数、合計発注額が正しく表示されていること。
2. **Given**: `ORD001` の発注データが存在する状態。
   **When**: `ORD001` の行を選択（または詳細ボタンをクリック）したとき。
   **Then**: 紐づく注文明細（MEM001, MEM002）の要員名、発注工数、発注単価、発注額（工数×単価）が別テーブルで正しく表示されていること。

---

### User Story 2 - 新規発注と注文明細の追加登録 (Priority: P1)

作業契約、発注先、年月を選択して新しい発注を作成し、そこに紐づく要員アサイン明細を追加登録できる。

**Why this priority**: 新規アサインやパートナーとの月次契約時に、動的に発注データを登録するために必須の機能。

**Independent Test**: 
新規登録フォームに正しい値を入力した際、発注IDが `ORD007` と自動採番され、追加した要員の単価から明細行の発注額が自動計算されて登録されること。

**Acceptance Scenarios**:

1. **Given**: 案件作業明細（アサイン明細）として `WK001`、発注先として `BP001`、年月として `2026-08-01` が選択可能な状態。
   **When**: 開始期間 of 重なりチェックおよび重複登録防止チェック（同一の `作業契約ID` + `年月` + `発注先ID`）を通過する年月を入力し、新規登録を実行したとき。
   **Then**: 注文IDが `ORD007` として新規登録され、合計工数 0.0、合計発注額 0 円の空の発注レコードが作成されること。
2. **Given**: 作成された `ORD007` の発注詳細画面が開いている状態。
   **When**: 発注先 `BP001` に所属する要員 `MEM001` を選択し、発注工数 `0.8` 人月を入力して明細行を追加したとき。
   **Then**: `MEM001` の要員単価 `1,000,000円` に基づき、発注額が `800,000円` として自動算出され、親発注の合計工数が `0.8` 人月、合計発注額が `800,000円` に自動更新されること。

---

### User Story 3 - 発注情報の編集・更新 (Priority: P2)

登録済みの発注データおよび注文明細（工数の増減、明細の追加・削除）を編集・更新できる。

**Why this priority**: 月次契約内容の調整や稼働工数の変更に伴う発注額の修正に対応するために必要。

**Independent Test**: 
既存の `ORD001` の明細行のうち、`MEM001` の工数を `0.8` から `0.5` に編集して保存した際、合計工数が `1.0` 人月、合計発注額が `850,000円` に再計算され保存されること。

**Acceptance Scenarios**:

1. **Given**: `ORD001` の発注編集画面が開いている状態。
   **When**: `MEM001` の発注工数を `0.8` から `0.5` に変更し、保存したとき。
   **Then**: 親発注データの `合計工数` が `1.0` 人月（0.5 + 0.5）、`合計発注額` が `850,000円`（0.5*100万 + 0.5*70万）に再計算されて永続化されること。
2. **Given**: `ORD001` の発注編集画面が開いている状態。
   **When**: `MEM002` の明細行を削除して保存したとき。
   **Then**: `ORD001` の合計工数が `0.8` 人月（MEM001分のみ）、合計発注額が `800,000円` になり、明細が1件のみ更新保存されること。

---

### User Story 4 - 発注データの物理削除 (Priority: P2)

不要になった発注データを物理削除する。発注データが削除された場合、紐づく注文明細データも連動してカスケード削除される。

**Why this priority**: 誤入力された発注データのクリーンアップを行うために必要。

**Independent Test**: 
実績の存在しない発注データを削除した際、一覧から消去され、リポジトリ上からも消滅すること。

**Acceptance Scenarios**:

1. **Given**: `ORD006` の発注データが存在する状態。
   **When**: 削除ボタンをクリックし、確認ダイアログに「OK」と答えたとき。
   **Then**: `ORD006` の発注および紐づいていた `MEM003`, `MEM004` の注文明細レコードがリポジトリから物理削除され、一覧から表示が消えること。

---

### Edge Cases

- **年月の期間検証制約**:
  - 発注登録時の `年月` (日付型, YYYY-MM-01) は、選択された `作業契約ID` で紐づく `案件作業明細` の期間（開始日〜終了日）と1日以上重なる期間が存在しなければならない。重なりがない場合は、保存時に `選択された年月は作業契約の対象期間外です。` エラーで登録がブロックされること。
- **発注工数の範囲制約**:
  - 注文明細の `発注工数` は `0以上1以下`（0.0 〜 1.0人月）、かつ小数点以下1桁まで。範囲外（例: `1.5` や `-0.1`）、あるいは小数点以下2桁以上の値が入力された場合は、明細追加時にエラー表示されブロックされること。

## 3. Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 注文IDは一意であり、システムにより自動採番（`ORD001` から始まる連番）されること。
- **FR-002**: 発注を作成する際、`作業契約ID`、`発注先ID`、`年月` は必須入力であること。
- **FR-003**: 発注の `年月` は `YYYY-MM-01` 形式で保持され、対応するアサイン期間（開始日〜終了日）と重なる期間があること。また、同一の `作業契約ID`・`年月`・`発注先ID` の組み合わせでの重複登録はエラーとし、登録を拒否すること。
- **FR-004**: 注文明細に要員を追加する際、選択可能な要員は指定した `発注先`（所属会社）に属する要員（要員マスタの `所属会社ID` が `発注先ID` と一致するもの）に限定されること。ドメインおよびアプリケーション層の保存時にも、要員の `所属会社ID` と発注の `発注先ID` が完全一致していることを厳密に検証し、不一致の場合はエラーで保存を拒否すること。
- **FR-005**: 注文明細の `発注工数` は 0以上1以下の数値、小数点以下1桁でなければならないこと。
- **FR-006**: 注文明細の `発注単価` は、選択した要員マスタに登録されている `単価` を自動適用すること。
- **FR-007**: 注文明細の `発注額` は `発注工数` × `発注単価` で動的に自動計算されること。
- **FR-008**: 発注データの `合計工数` および `合計発注額` は、常に配下の `注文明細` の集計値が動的に反映されること。
- **FR-009**: 発注データを削除した際、その発注に紐づくすべての `注文明細` も同時に自動削除（カスケード削除）されること。

### Key Entities

- **発注 (PartnerOrder)**:
  - 役割：発注データの集約ルート。発注先、作業契約、発注年月を束ねる。
  - 主な属性：
    - 注文ID (`id`) : 一意キー
    - 作業契約ID (`caseAssignmentId`) : 関連アサインID
    - 発注先ID (`partnerId`) : パートナーID
    - 年月 (`targetMonth`) : 発注対象年月（日付表現 `YYYY-MM-01`）
    - 合計工数 (`totalEffort`) : 導出値（注文明細の発注工数合計）
    - 合計発注額 (`totalAmount`) : 導出値（注文明細の発注額合計）
- **注文明細 (OrderDetail)**:
  - 役割：発注に紐づく要員アサインの明細情報。
  - 主な属性：
    - 注文ID (`orderId`) : 親発注ID（複合キー）
    - 要員ID (`staffId`) : 要員ID（複合キー）
    - 発注工数 (`orderEffort`) : 実働工数（人月、0.0〜1.0）
    - 発注単価 (`orderPrice`) : 導出値（要員の単価）
    - 年月 (`targetMonth`) : 導出値（親発注の年月）
    - 発注先ID (`partnerId`) : 導出値（親発注の発注先ID）
    - 発注額 (`orderAmount`) : 導出値（発注工数 × 発注単価）

---

## 4. Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: パートナー要員の追加および工数の書き換え時に、発注単価から明細の `発注額`、および親発注の `合計工数`・`合計発注額` がミリ秒単位で瞬時に再計算されて画面上に表示されること。
- **SC-002**: 不正な年月入力（作業契約期間外）や、不正な工数値（1.0を超える値など）を入力した際、保存前にエラーで即時ブロックされ、不正データがリポジトリに書き込まれないこと。
- **SC-003**: 登録した発注情報（合計発注額）が、案件作業アサイン側の製造原価計算へ正しくリアルタイムに反映され、アサイン明細の粗利率の計算結果に反映されること。

---

## 5. Assumptions

- **要員マスタデータ**: `MEM001`〜`MEM004` の要員は、F04「要員マスタ」が事前に正しく登録されていることを前提とする。
- **発注先データ**: `BP001`〜`BP002` のパートナーは、F03「発注先マスタ」が事前に登録されていることを前提とする。
- **期間検証**: 開始日・終了日を伴う作業契約データは、F06「案件作業契約明細」のデータが存在することを前提とする。
- **データの永続化**: すべてのデータ変更は、憲法に従ってブラウザの `LocalStorage` に自動永続化され、リロード時にも完全に状態が復元される。
