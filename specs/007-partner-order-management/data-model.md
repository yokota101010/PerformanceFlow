# Data Model: 発注（注文）管理

ドメインモデルの正本である `.specify/memory/domain-model.md` に基づき、本機能が取り扱う集約およびデータ構造を以下に定義する。

---

## 1. 状態遷移図 (State Transition)

発注（注文）データの作成から明細構成、および物理削除までのライフサイクルを示す。

```mermaid
stateDiagram-RU
    [*] --> 新規作成 : 登録操作開始
    新規作成 --> 空の発注 : 保存 (作業契約・発注先・年月指定)
    空の発注 --> 明細登録済 : 注文明細追加 (要員・工数登録)
    明細登録済 --> 明細登録済 : 注文明細追加/変更/削除 (自動再計算)
    明細登録済 --> [*] : 物理削除 (カスケード削除)
    空の発注 --> [*] : 物理削除 (カスケード削除)
```

---

## 2. テーブル定義

### テーブル: `発注`
* 集約における役割: 👑 **集約ルート (Aggregate Root)**
* 物理格納キー: `performance_flow_partner_orders` (LocalStorage)

| カラム名 (物理) | データ型 | PK | FK | UQ | 区分 | カラム定義・ビジネスルール |
| :--- | :--- | :---: | :---: | :---: | :--- | :--- |
| `注文ID` | `VARCHAR(255)` | ✅ |  |  | 独立 | 形式：`ORDnnn`（ORD + 3桁連番） |
| `作業契約ID` | `VARCHAR(255)` |  | ✅ (`案件作業明細.作業契約ID`) | ✅ (UQ1) | 独立 |  |
| `発注先ID` | `VARCHAR(255)` |  | ✅ (`発注先.発注先ID`) | ✅ (UQ1) | 独立 |  |
| `年月` | `DATE` |  |  | ✅ (UQ1) | 独立 | - YYYY-MM-01形式。<br />- `作業契約ID` で紐づく `案件作業明細` の `開始日`〜`終了日` と重なる期間があること。 |
| `合計工数` | `DECIMAL` |  |  |  | 導出項目 | * 導出式: 同一 `注文ID` の `注文明細` の `発注工数` の合計を設定。 |
| `合計発注額` | `INT` |  |  |  | 導出項目 | * 導出式: 同一 `注文ID` の `注文明細` の `発注額` の合計を設定。 |

### テーブル: `注文明細`
* 集約における役割: 集約メンバー
* 物理格納キー: `performance_flow_partner_order_details` (LocalStorage または `発注` 配下にシリアライズして保持)

| カラム名 (物理) | データ型 | PK | FK | UQ | 区分 | カラム定義・ビジネスルール |
| :--- | :--- | :---: | :---: | :---: | :--- | :--- |
| `注文ID` | `VARCHAR(255)` | ✅ | ✅ (`発注.注文ID`) |  | 独立 |  |
| `要員ID` | `VARCHAR(255)` | ✅ | ✅ (`要員.要員ID`) |  | 独立 | 下記2項目の値が等しい必要がある。<br />- `要員` の `所属会社ID`<br />- `発注` の `発注先ID` |
| `発注工数` | `DECIMAL` |  |  |  | 独立 | - 単位は人月。<br />- 0以上1以下の値を取る。<br />- 小数点以下の桁数は1桁。 |
| `発注単価` | `INT` |  |  |  | 導出項目 | * 導出式: `要員` テーブルの `単価` を設定。 |
| `年月` | `DATE` |  |  |  | 導出項目 | * 導出式: 親 `発注` の `年月` を設定。 |
| `発注先ID` | `VARCHAR(255)` |  |  |  | 導出項目 | * 導出式: 親 `発注` の `発注先ID` を設定。 |
| `発注額` | `INT` |  |  |  | 導出項目 | * 導出式: `発注工数` × `発注単価` を設定。 |
